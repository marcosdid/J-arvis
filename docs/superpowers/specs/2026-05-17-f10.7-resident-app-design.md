# F10.7 spec #1 — Resident App (design)

> **Status:** design (brainstorm 2026-05-17). Primeira de duas specs do bloco
> F10.7 (OS integration). Esta cobre: system tray icon, single-instance
> D-Bus via Wails nativo, close-to-tray lifecycle, e CLI `--focus` flag.
> Spec #2 (próxima) tratará drag-folder → create project.

## Problema

O J-arvis hoje roda como uma janela Wails comum: fechar o X mata o processo
(perdendo master Claude session, sessões ai-jail ativas, runs Docker em curso).
Não tem presence no system tray, então o usuário precisa lembrar de uma janela
flutuante específica entre dezenas. Não tem atalho global pra trazer foco —
precisa Alt+Tab via toda a stack de janelas.

Spec #1 transforma o J-arvis num **app residente**: ícone fixo no tray, fechar
a janela esconde mas não mata, atalho do DE (Super+J via bind manual) traz
foco, e segunda invocação de `jarvis` foca a primeira instância em vez de
spawnar duplicado.

## Não-objetivos

- Drag-folder de file manager → cria project (vai pro spec #2).
- Hotkey global in-process via libkeybinder3 cgo (decidido: bind manual no DE
  chamando `jarvis --focus`, paridade cross-DE sem cgo).
- Dynamic tooltip "N sessions, M awaiting" (YAGNI; logs/UI bastam).
- Menu "Nova task..." (YAGNI; user abre UI e clica).
- Status-aware icon (idle/executing/awaiting) — polish pra depois.
- Theme-aware icon (light/dark swap) — polish pra depois.
- D-Bus notifications custom `org.freedesktop.Notifications` — Wails
  `runtime.MessageDialog` cobre o pouco que precisa hoje.
- Autostart `.desktop` em `~/.config/autostart` — fica pra F10.8 packaging.

## Decisões de design (do brainstorm)

| Pergunta | Resposta |
|---|---|
| Comportamento ao fechar janela | **Close-to-tray**: esconde janela, daemon segue rodando |
| Menu do tray | **Mínimo**: "Mostrar janela" + "Quit"; tooltip estático "J-arvis" |
| Atalho global | **Bind manual no DE**: docs explicam GNOME/KDE/Sway bindando Super+J → `jarvis --focus` |
| Fallback sem D-Bus/tray | **Fail-open**: sem tray, sem single-instance; app roda como instância isolada |
| Assets do ícone | **PNG embedado** via `//go:embed`, 1 ícone idle (32x32) |

## Infraestrutura aproveitada do Wails v2

Investigação confirmou que **Wails v2.12.0 já tem dois dos quatro
subsistemas nativos**, eliminando código D-Bus custom:

| Feature | Wails v2 API | Linux backend |
|---|---|---|
| Single-instance | `options.App.SingleInstanceLock{UniqueId, OnSecondInstanceLaunch}` | D-Bus session bus + `RequestName` + `Export` (em `internal/frontend/desktop/linux/single_instance.go`) |
| Close-to-tray | `options.App.HideWindowOnClose: true` | Wails intercepta GTK close signal |
| Drag-folder | `options.App.DragAndDrop{EnableFileDrop, CSSDropProperty, CSSDropValue}` | (spec #2) |

**Único subsistema externo:** o system tray. Wails v2 não tem tray nativo
(Wails v3 adiciona, mas não usamos). Biblioteca canônica: **`fyne.io/systray`**
(fork mantido do `getlantern/systray`, mesma API, AppIndicator-based).

## Fluxos end-to-end

### Startup (primeira instância)

```
$ jarvis [--focus]
│
├─ Parse os.Args (osintegration/cli.go) → CLIFlags{Focus: bool}
├─ DetectTray() → bool   (checa $DISPLAY ou $WAYLAND_DISPLAY)
├─ store.Open + Migrate + repos + services + APIs (como hoje)
│
├─ Configura options.App:
│  ├─ Bind: [...] (como hoje)
│  ├─ HideWindowOnClose: trayAvailable
│  └─ SingleInstanceLock: se trayAvailable: &{UniqueId, OnSecondInstanceLaunch}
│
└─ wails.Run(opts) — bloqueia até OnShutdown ou erro
   └─ OnStartup(ctx):
      ├─ app.startup(ctx); realBus.Store(emitter)         // existente
      ├─ appCtx = ctx                                      // NOVO: captura
      └─ if trayAvailable { go osintegration.StartTray(ctx, onShow, onQuit) }
```

`onShow` e `onQuit` são closures construídos em main.go:

```go
onShow := func() {
    runtime.WindowShow(appCtx)
    runtime.WindowUnminimise(appCtx)
}
onQuit := func() {
    runtime.Quit(appCtx)  // dispara Wails OnShutdown chain existente
}
```

### Tray init (dentro do StartTray goroutine)

```go
func StartTray(ctx context.Context, onShow, onQuit func()) {
    lib := newRealTrayLib()  // adapter sobre fyne.io/systray
    if err := lib.RunWithExternalLoop(onReady(ctx, lib, onShow, onQuit), onExit(onQuit)); err != nil {
        log.Printf("tray init failed, running headless: %v", err)
        return  // fail-open
    }
}

func onReady(ctx, lib, onShow, onQuit) func() {
    return func() {
        lib.SetIcon(trayIconPNG)
        lib.SetTooltip("J-arvis")
        mShow := lib.AddMenuItem("Mostrar janela", "")
        lib.AddSeparator()
        mQuit := lib.AddMenuItem("Quit", "Encerrar J-arvis (mata sessions, runs, master)")

        go func() {
            for {
                select {
                case <-ctx.Done():
                    return
                case <-mShow.Clicked():
                    onShow()
                case <-mQuit.Clicked():
                    if !quitInProgress.Swap(true) {
                        lib.Quit()
                        onQuit()
                    }
                    return
                }
            }
        }()
    }
}
```

`quitInProgress` é `atomic.Bool` package-level, garante idempotência sob
clicks rápidos.

### Close-to-tray (usuário clica X)

- `HideWindowOnClose: true` → Wails intercepta GTK close signal, chama
  `WindowHide(ctx)` em vez de quit.
- Daemon segue rodando: master Claude (PID em `master_session` table),
  sessões ai-jail (rows em `sessions`), containers Docker (rows em
  `run_instances`), watchers de bootstrap (in-memory map), localhttp listener.
- Tray icon permanece visível. Único caminho pra encerrar de verdade: tray
  "Quit".

### Tray "Quit"

```
[user clica menu Quit]
│
├─ ClickedCh recebe
├─ quitInProgress.Swap(true) → false (1ª vez), proceed
├─ lib.Quit()                  // systray remove ícone do tray
├─ onQuit() → runtime.Quit(appCtx)
└─ Wails OnShutdown chain (existente):
   ├─ masterSvc.Stop(ctx)      // kill master Claude PID
   └─ localSrv.Stop()          // para listener 127.0.0.1:N
```

### Segunda invocação (Super+J ou `jarvis` em outro terminal)

```
$ jarvis --focus    (ou apenas `jarvis`)
│
├─ Wails SingleInstanceLock claim:
│  ├─ RequestName(D-Bus) → NameAlreadyTaken
│  ├─ Marshal SecondInstanceData{Args: ["--focus"], WorkingDirectory}
│  ├─ Call(1st instance D-Bus path, "SendMessage", serialized)
│  └─ os.Exit(1)
│
└─ (Primeiro proc receives via SecondInstanceLock callback):
   └─ handleSecondInstance(data):
      ├─ runtime.WindowShow(appCtx)
      └─ runtime.WindowUnminimise(appCtx)
```

`--focus` é semântica documentada — o handler **ignora os args** e sempre
foca (consistente com "second invoke = front"). Args ficam disponíveis pra
futuro: `--task=<id>` poderia navegar pra task X.

## Detect-then-configure

```go
// main.go (antes de configurar options.App)
trayAvailable := osintegration.DetectTray()

appOpts := &options.App{
    Title:           "J-arvis",
    Width:           1400,
    Height:          900,
    Bind:            []any{...},
    OnStartup:       startupFn,
    OnShutdown:      shutdownFn,
    // ... outros campos existentes
}

if trayAvailable {
    appOpts.HideWindowOnClose = true
    appOpts.SingleInstanceLock = &options.SingleInstanceLock{
        UniqueId:               "com.marcosdid.jarvis",
        OnSecondInstanceLaunch: handleSecondInstance,
    }
}
```

`DetectTray()` pragmático na 1ª versão:

```go
func DetectTray() bool {
    return os.Getenv("DISPLAY") != "" || os.Getenv("WAYLAND_DISPLAY") != ""
}
```

Headless (CI, container minimal sem display): retorna false → app roda sem
tray, sem single-instance, close button = quit normal. Não vira zumbi.

## Tratamento de erros

| Cenário | Comportamento |
|---|---|
| `DetectTray()` retorna false | Skip todo o tray + single-instance setup. App roda standalone. |
| `systray.RunWithExternalLoop` retorna erro (raro com display set mas tray indisponível) | Log warning, sem panic. Wails continua. Se `HideWindowOnClose` ativo → app pode virar zumbi (ver mitigação abaixo). |
| D-Bus session bus indisponível com display set (improvável) | Wails `SingleInstanceLock.SetupSingleInstance` retorna silenciosamente sem claim. Segunda instância roda em paralelo. |
| `runtime.WindowShow` em ctx cancelado | Wails retorna erro silencioso. Log debug. |
| User clica Quit 2x rápido | `atomic.Bool` swap garante 1 cleanup. |
| `os.Args` vazio (impossível) | `ParseFlags` retorna zero value, OK. |

**Mitigação zumbi:** se tray loop morre sem ter sido por click "Quit",
`onExit` callback dispara `runtime.Quit(appCtx)` automaticamente. Isso cobre
o caso "tray crashou no meio do uso" mas não cobre "tray nunca subiu E
DetectTray foi false-positive". Solução de raiz: confiar em DetectTray;
mitigação adicional: `onExit` chama `runtime.Quit` se `quitInProgress` ainda
é false (loop morreu sem o user pedir):

```go
onExit := func(onQuit func()) func() {
    return func() {
        if !quitInProgress.Swap(true) {
            onQuit()  // runtime.Quit
        }
    }
}
```

## CLI

```go
type CLIFlags struct {
    Focus bool   // --focus
}

func ParseFlags() CLIFlags {
    var f CLIFlags
    for _, arg := range os.Args[1:] {
        switch arg {
        case "--focus":
            f.Focus = true
        }
    }
    return f
}
```

Parsing manual (não `flag.Parse`) pra evitar conflito com Wails CLI runner
e ficar trivial de mockar. Args desconhecidos ignorados silenciosamente
(forward compat).

## Riscos

| Risco | Mitigação |
|---|---|
| GNOME 40+ não mostra system tray sem extension `AppIndicator and KStatusNotifierItem Support` | Documentado em `docs/os-integration/tray-setup.md` |
| Wayland + compositors raw (Sway pure, raw Wayland) não suportam StatusNotifierItem | Fail-open documentado |
| `fyne.io/systray` requer main thread no Linux mas Wails também — depende de `RunWithExternalLoop` API | Validar na Stage 1 do plano (smoke build + run em DE real). Se quebrar, escalar |
| `HideWindowOnClose` + tray failure → zumbi | `DetectTray` antes de aplicar; `onExit` fallback dispara `runtime.Quit` |
| `SingleInstanceLock.UniqueId` colide com app de terceiros | Namespace explícito `com.marcosdid.jarvis` |
| Dev tem 2 jarvis no PATH com mesma UniqueId | Mesma UniqueId → 2ª chama focus da 1ª. **É a feature** |
| `runtime.WindowShow` em janela já visível | No-op (Wails handles) |
| Quit acontece mid-write do master Claude | `OnShutdown` chain existente já lida (`masterSvc.Stop` antes do `localSrv.Stop`); spec não muda essa ordem |
