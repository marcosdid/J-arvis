# F10.7 spec #1 — Resident App (design)

> **Status:** design (brainstorm 2026-05-17). Primeira de duas specs do bloco
> F10.7 (OS integration). Esta cobre: system tray icon, single-instance
> D-Bus via Wails nativo, close-to-tray lifecycle, e CLI `--focus` flag.
> Spec #2 (próxima) tratará drag-folder → create project.

## Problema

O J-arvis hoje roda como uma janela Wails comum: fechar o X mata o processo
(perdendo master Claude session, sessões ai-jail ativas, runs Docker em curso).
Não tem presence no system tray, então o usuário precisa lembrar de uma janela
flutuante específica entre dezenas. Não tem atalho global pra trazer foco —
precisa Alt+Tab via toda a stack de janelas.

Spec #1 transforma o J-arvis num **app residente**: ícone fixo no tray, fechar
a janela esconde mas não mata, atalho do DE (Super+J via bind manual) traz
foco, e segunda invocação de `jarvis` foca a primeira instância em vez de
spawnar duplicado.

## Não-objetivos

- Drag-folder de file manager → cria project (vai pro spec #2).
- Hotkey global in-process via libkeybinder3 cgo (decidido: bind manual no DE
  chamando `jarvis --focus`, paridade cross-DE sem cgo).
- Dynamic tooltip "N sessions, M awaiting" (YAGNI; logs/UI bastam).
- Menu "Nova task..." (YAGNI; user abre UI e clica).
- Status-aware icon (idle/executing/awaiting) — polish pra depois.
- Theme-aware icon (light/dark swap) — polish pra depois.
- D-Bus notifications custom `org.freedesktop.Notifications` — Wails
  `runtime.MessageDialog` cobre o pouco que precisa hoje.
- Autostart `.desktop` em `~/.config/autostart` — fica pra F10.8 packaging.

## Decisões de design (do brainstorm)

| Pergunta | Resposta |
|---|---|
| Comportamento ao fechar janela | **Close-to-tray**: esconde janela, daemon segue rodando |
| Menu do tray | **Mínimo**: "Mostrar janela" + "Quit"; tooltip estático "J-arvis" |
| Atalho global | **Bind manual no DE**: docs explicam GNOME/KDE/Sway bindando Super+J → `jarvis --focus` |
| Fallback sem D-Bus/tray | **Fail-open**: sem tray, sem single-instance; app roda como instância isolada |
| Assets do ícone | **PNG embedado** via `//go:embed`, 1 ícone idle (32x32) |

## Infraestrutura aproveitada do Wails v2

Investigação confirmou que **Wails v2.12.0 já tem dois dos quatro
subsistemas nativos**, eliminando código D-Bus custom:

| Feature | Wails v2 API | Linux backend |
|---|---|---|
| Single-instance | `options.App.SingleInstanceLock{UniqueId, OnSecondInstanceLaunch}` | D-Bus session bus + `RequestName` + `Export` (em `internal/frontend/desktop/linux/single_instance.go`) |
| Close-to-tray | `options.App.HideWindowOnClose: true` | Wails intercepta GTK close signal |
| Drag-folder | `options.App.DragAndDrop{EnableFileDrop, CSSDropProperty, CSSDropValue}` | (spec #2) |

**Único subsistema externo:** o system tray. Wails v2 não tem tray nativo
(Wails v3 adiciona, mas não usamos). Biblioteca canônica: **`fyne.io/systray`**
(fork mantido do `getlantern/systray`, mesma API, AppIndicator-based).

### Interface abstrata sobre `fyne.io/systray`

A API real do `fyne.io/systray` é:

```go
// fyne.io/systray
func RunWithExternalLoop(onReady, onExit func()) (start, end func())
func SetIcon(data []byte)
func SetTooltip(t string)
func AddMenuItem(title, tooltip string) *MenuItem
func AddSeparator()
func Quit()
// MenuItem expõe ClickedCh chan struct{}
```

`RunWithExternalLoop` **não retorna erro** — só registra callbacks e devolve
duas funções `start` (chama `nativeStart` — registra D-Bus path) e `end`
(chama `nativeEnd` + `Quit` — desregistra). D-Bus connection failures dentro
do `nativeStart` são apenas logadas (`systray_unix.go:194`). Por isso a
detecção de fail-open precisa ser **pre-flight** (probe explícito do D-Bus
session bus antes de chamar `start`), não retroativa.

Abstração nossa pra testes:

```go
// internal/osintegration/tray.go
type trayItem interface {
    Clicked() <-chan struct{}
}

type trayLib interface {
    SetIcon(data []byte)
    SetTooltip(t string)
    AddMenuItem(title, tooltip string) trayItem
    AddSeparator()
    Quit()
}

// Factory: cria o lib + devolve start/end controlados pelo caller.
type trayFactory func(onReady, onExit func()) (lib trayLib, start, end func())
```

Produção: thin adapter wrapping `fyne.io/systray`. Tests: `fakeTrayFactory`
retorna um `fakeTrayLib` + `start`/`end` que apenas marcam invocações.

## Fluxos end-to-end

### Startup (primeira instância)

```
$ jarvis [--focus]
│
├─ Parse os.Args (osintegration/cli.go) → CLIFlags{Focus: bool}
├─ DetectTray() → bool   (checa $DISPLAY ou $WAYLAND_DISPLAY)
├─ store.Open + Migrate + repos + services + APIs (como hoje)
│
├─ Configura options.App:
│  ├─ Bind: [...] (como hoje)
│  ├─ HideWindowOnClose: trayAvailable
│  └─ SingleInstanceLock: se trayAvailable: &{UniqueId, OnSecondInstanceLaunch}
│
└─ wails.Run(opts) — bloqueia até OnShutdown ou erro
   └─ OnStartup(ctx):
      ├─ app.startup(ctx); realBus.Store(emitter)         // existente
      ├─ appCtx = ctx                                      // NOVO: captura
      └─ if trayAvailable { go osintegration.StartTray(ctx, onShow, onQuit) }
```

`onShow` e `onQuit` são closures construídos em main.go:

```go
onShow := func() {
    runtime.WindowShow(appCtx)
    runtime.WindowUnminimise(appCtx)
}
onQuit := func() {
    runtime.Quit(appCtx)  // dispara Wails OnShutdown chain existente
}
```

### Tray controller (lifecycle owned by main.go)

Em vez de uma função `StartTray` que esconde o lifecycle, exponhamos um
`TrayController` cujo lifecycle (`Start`, `Stop`) o `main.go` controla via
`OnStartup` / `OnShutdown`. Isso encaixa naturalmente com o `start`/`end`
do `RunWithExternalLoop` e injeta `quitInProgress` como campo (não package
state — testes ficam isolados).

```go
// internal/osintegration/tray.go
type TrayController struct {
    factory        trayFactory
    onShow         func()
    onQuit         func()
    start, end     func()
    lib            trayLib
    quitInProgress atomic.Bool
}

func NewTrayController(onShow, onQuit func()) *TrayController {
    return &TrayController{factory: realTrayFactory, onShow: onShow, onQuit: onQuit}
}

// PreflightOK probes D-Bus session bus + display env. Returns true if the
// environment can host a tray. Called by main.go BEFORE configuring
// HideWindowOnClose/SingleInstanceLock.
func PreflightOK() bool {
    if os.Getenv("DISPLAY") == "" && os.Getenv("WAYLAND_DISPLAY") == "" {
        return false
    }
    conn, err := dbus.SessionBus()
    if err != nil {
        return false
    }
    _ = conn.Close()  // probe only; release back to bus
    return true
}

// Start registers callbacks + invokes the tray library's start function.
// Must be called from OnStartup AFTER ctx is captured. Idempotent.
func (t *TrayController) Start(ctx context.Context) {
    onReady := t.makeOnReady(ctx)
    onExit := t.makeOnExit()
    t.lib, t.start, t.end = t.factory(onReady, onExit)
    if t.start != nil {
        t.start()
    }
}

// Stop is called by main.go from OnShutdown. Tears down the tray icon
// deterministically (vs relying on internal Quit chain).
func (t *TrayController) Stop() {
    if t.end != nil {
        t.end()  // runs nativeEnd + Quit() on the lib
    }
}

func (t *TrayController) makeOnReady(ctx context.Context) func() {
    return func() {
        t.lib.SetIcon(trayIconPNG)
        t.lib.SetTooltip("J-arvis")
        mShow := t.lib.AddMenuItem("Mostrar janela", "")
        t.lib.AddSeparator()
        mQuit := t.lib.AddMenuItem("Quit", "Encerrar J-arvis (mata sessions, runs, master)")

        go func() {
            for {
                select {
                case <-ctx.Done():
                    return
                case <-mShow.Clicked():
                    t.onShow()
                case <-mQuit.Clicked():
                    // STEADY-STATE PATH: user clicked Quit. Win the swap;
                    // onExit (which runs after lib.Quit cascades) finds
                    // quitInProgress=true and no-ops.
                    if !t.quitInProgress.Swap(true) {
                        t.lib.Quit()
                        t.onQuit()
                    }
                    return
                }
            }
        }()
    }
}

func (t *TrayController) makeOnExit() func() {
    return func() {
        // FALLBACK PATH: lib loop ended without user clicking Quit (e.g.,
        // D-Bus disconnect mid-session). Win the swap to trigger graceful
        // shutdown. In the steady-state path, mQuit handler already swapped
        // and this is a no-op.
        if !t.quitInProgress.Swap(true) {
            t.onQuit()
        }
    }
}
```

**Ownership contract:**

The `fyne.io/systray` library, in `RunWithExternalLoop` mode, **only invokes
`onExit` when our `end()` function is called** (which we map to
`TrayController.Stop()`, invoked from Wails `OnShutdown`). `lib.Quit()` by
itself only closes the internal `quitChan` to terminate the lib's goroutine
— it does NOT cascade through `nativeEnd → onExit` in external-loop mode.
This shapes the two real paths:

- **Path A (steady-state)** — user clicks tray "Quit":
  1. `mQuit.Clicked()` fires
  2. `quitInProgress.Swap(true)` returns `false` → swap wins
  3. `t.lib.Quit()` → closes `quitChan`, stops lib goroutine
  4. `t.onQuit()` → `runtime.Quit(appCtx)` → fires Wails `OnShutdown`
  5. Wails `OnShutdown` calls `trayCtl.Stop()` → `end()` → `nativeEnd` →
     `runSystrayExit` → our `onExit` → `Swap(true)` returns `true` → no-op
  6. Existing `OnShutdown` chain (masterSvc.Stop, localSrv.Stop) continues
  - Net: `onQuit()` called exactly once.

- **Path B (shutdown without tray click)** — process receives SIGTERM,
  user closes via `kill`, or `runtime.Quit` fires from some other source
  (e.g., future menu action):
  1. Wails `OnShutdown` calls `trayCtl.Stop()` → `end()` →
     `nativeEnd → onExit`
  2. `quitInProgress.Swap(true)` returns `false` → swap wins
  3. `t.onQuit()` → `runtime.Quit(appCtx)` — already in shutdown, but Wails
     handles idempotently (verified — `Quit` re-entry is safe)
  - Net: `onQuit()` called exactly once.

- **NOT a covered path:** D-Bus disconnect mid-session. In external-loop
  mode, `onExit` is shutdown-only; the library has no callback for runtime
  bus loss. If D-Bus dies after `nativeStart` succeeded, the tray icon
  silently disappears but the app keeps running normally. User can still
  use the window (which is visible — close-to-tray only hides on user X
  click). If they then click X, app enters zumbi state (no tray to reopen).
  **Mitigation:** documented as known limitation in
  `docs/os-integration/tray-setup.md`; if D-Bus is flaky on user's setup,
  they should disable `HideWindowOnClose` (future env flag). YAGNI for
  detection now — extremely rare on stable distros.

Exactly one `onQuit()` per process lifetime, guaranteed by `atomic.Bool.Swap`.

### Close-to-tray (usuário clica X)

- `HideWindowOnClose: true` → Wails intercepta GTK close signal, chama
  `WindowHide(ctx)` em vez de quit.
- Daemon segue rodando: master Claude (PID em `master_session` table),
  sessões ai-jail (rows em `sessions`), containers Docker (rows em
  `run_instances`), watchers de bootstrap (in-memory map), localhttp listener.
- Tray icon permanece visível. Único caminho pra encerrar de verdade: tray
  "Quit".

### Tray "Quit"

```
[user clica menu Quit]
│
├─ ClickedCh recebe
├─ quitInProgress.Swap(true) → false (Path A vence)
├─ lib.Quit()                       // fecha quitChan interno do systray;
│                                   // NÃO cascateia nativeEnd em external-loop mode
├─ onQuit() → runtime.Quit(appCtx)
│
└─ Wails OnShutdown chain dispara em ordem:
   ├─ trayCtl.Stop()                // NOVO: chama end() → nativeEnd →
   │                                // runSystrayExit → onExit → swap finds
   │                                // true → no-op (Path A já venceu)
   ├─ masterSvc.Stop(ctx)           // existente: kill master Claude PID
   └─ localSrv.Stop()               // existente: para listener 127.0.0.1:N
```

`trayCtl.Stop()` precede `masterSvc.Stop` no `OnShutdown` (símile à
existente "stop master FIRST" rationale: a operação de UI deve sumir antes
das operações de domínio). É o caller (`main.go`) que ordena.

### Segunda invocação (Super+J ou `jarvis` em outro terminal)

```
$ jarvis --focus    (ou apenas `jarvis`)
│
├─ Wails SingleInstanceLock claim:
│  ├─ RequestName(D-Bus) → NameAlreadyTaken
│  ├─ Marshal SecondInstanceData{Args: ["--focus"], WorkingDirectory}
│  ├─ Call(1st instance D-Bus path, "SendMessage", serialized)
│  └─ os.Exit(1)
│
└─ (Primeiro proc receives via SecondInstanceLock callback):
   └─ handleSecondInstance(data):
      ├─ runtime.WindowShow(appCtx)
      └─ runtime.WindowUnminimise(appCtx)
```

`--focus` é semântica documentada — o handler **ignora os args** e sempre
foca (consistente com "second invoke = front"). Args ficam disponíveis pra
futuro: `--task=<id>` poderia navegar pra task X.

## Detect-then-configure

```go
// main.go (antes de configurar options.App)
trayAvailable := osintegration.PreflightOK()

trayCtl := osintegration.NewTrayController(onShow, onQuit)

appOpts := &options.App{
    Title:      "J-arvis",
    Width:      1400,
    Height:     900,
    Bind:       []any{...},
    OnStartup:  startupFn,    // capturará ctx + chamará trayCtl.Start(ctx) se trayAvailable
    OnShutdown: shutdownFn,   // chamará trayCtl.Stop() PRIMEIRO se trayAvailable
    // ... outros campos existentes
}

if trayAvailable {
    appOpts.HideWindowOnClose = true
    appOpts.SingleInstanceLock = &options.SingleInstanceLock{
        UniqueId:               "com.marcosdid.jarvis",
        OnSecondInstanceLaunch: handleSecondInstance,
    }
}
```

`PreflightOK()` faz duas checagens (declarado na seção de interface acima):

1. **Display env** — `$DISPLAY != ""` OU `$WAYLAND_DISPLAY != ""`. Pega CI
   headless e containers minimais.
2. **D-Bus session bus probe** — `dbus.SessionBus()` tenta conectar; falha
   se não há `DBUS_SESSION_BUS_ADDRESS` ou se o daemon morreu. Probe
   imediato (sem espera). **Não chamamos `conn.Close()`** porque
   `SessionBus()` retorna a conexão cacheada (package var protegida por
   mutex em godbus); fechá-la invalidaria o cache e forçaria reconnect em
   `nativeStart` mais tarde, além de potencialmente racear com outros
   consumidores do bus no processo. Sucesso == cache válido pra resto do
   processo.

Se ambos passam → fail-open desativado (configura tray + single-instance);
caso contrário → fail-open (skip tudo, comportamento standalone).

**Limitação conhecida:** rodar via SSH com X11 forwarding (`ssh -X`) seta
`$DISPLAY` mas o tray remoto pode não ter StatusNotifierItem — `PreflightOK`
retorna true, `nativeStart` loga o erro de D-Bus interno, e o app fica
zumbi (close-to-tray ativo, mas sem tray pra reabrir). Mitigado pelo
fallback path do `onExit` (vide seção Tray controller acima: se o loop
morrer sem user click Quit, `runtime.Quit` dispara automaticamente).
Documentado em `docs/os-integration/tray-setup.md`.

## Tratamento de erros

| Cenário | Comportamento |
|---|---|
| `DetectTray()` retorna false | Skip todo o tray + single-instance setup. App roda standalone. |
| `nativeStart` D-Bus connect falha silenciosamente (PreflightOK false-positive raro com `ssh -X`) | `lib.SetIcon`/`SetTooltip` viram no-ops (instance.conn nil — lib já protege). Menu não aparece. Quando user fechar a janela, fica zumbi. `onExit` fallback (descrito acima) dispara `runtime.Quit` quando o loop morrer. |
| D-Bus session bus indisponível com display set (improvável) | Wails `SingleInstanceLock.SetupSingleInstance` retorna silenciosamente sem claim. Segunda instância roda em paralelo. |
| `runtime.WindowShow` em ctx cancelado | Wails retorna erro silencioso. Log debug. |
| User clica Quit 2x rápido | `atomic.Bool` swap garante 1 cleanup. |
| `os.Args` vazio (impossível) | `ParseFlags` retorna zero value, OK. |

**Mitigação zumbi:** a lógica está implementada no `makeOnExit` mostrado na
seção Tray controller acima. Resumo:

- Path A (steady-state — user clica menu Quit): `mQuit` handler vence o
  `atomic.Bool.Swap` → roda `lib.Quit()` → cascata interna chama
  `runSystrayExit` → chama nosso `onExit` → vê `quitInProgress=true` →
  no-op. Exatamente 1 `onQuit()` chamado.
- Path B (fallback — D-Bus crashou ou loop morreu sem user action):
  `onExit` ganha o swap → chama `onQuit()` → `runtime.Quit(appCtx)`.
- Path C (zumbi raro — `PreflightOK` foi false-positive `ssh -X`, tray
  nunca subiu, `nativeStart` retornou silenciosamente): tray loop nunca
  inicia → `onExit` nunca dispara → nenhum cleanup automático. Mitigação
  parcial: `TrayController.Stop()` (chamado por Wails `OnShutdown` em qualquer
  caminho de shutdown) faz `end()` → `nativeEnd` → `runSystrayExit` →
  `onExit` → swap path B → `runtime.Quit`. Loop redundante mas cobre.

## CLI

```go
type CLIFlags struct {
    Focus bool   // --focus
}

func ParseFlags() CLIFlags {
    var f CLIFlags
    for _, arg := range os.Args[1:] {
        switch arg {
        case "--focus":
            f.Focus = true
        }
    }
    return f
}
```

Parsing manual (não `flag.Parse`) pra evitar conflito com Wails CLI runner
e ficar trivial de mockar. Args desconhecidos ignorados silenciosamente
(forward compat).

### Unit — `internal/osintegration/tray_test.go`

Tests injetam `fakeTrayFactory` no `TrayController` e usam um
`TrayController` fresco por teste — `quitInProgress` é campo de instância
(não package state), então isolamento entre testes é automático.

**`fakeTrayFactory` contract:** o `end` retornado pelo fake DEVE invocar
o `onExit` que foi passado ao factory — isso espelha o que `nativeEnd` faz
em produção (`runSystrayExit → onExit`). Sem esse contrato, testes como
`TestTrayController_OnExitFiresOnQuitWhenSteadyStateDidNot` ficam
estruturalmente inválidos (chamam `Stop()` esperando cascata em `onExit`
que nunca chega). Esqueleto:

```go
func fakeTrayFactory(onReady, onExit func()) (lib trayLib, start, end func()) {
    lib = &fakeTrayLib{}
    start = func() { onReady() }
    end = func() { onExit() }  // mirrors nativeEnd → runSystrayExit
    return
}
```

|---|---|

| `TestPreflightOK_WithX11Display` | `t.Setenv("DISPLAY", ":0")` — se D-Bus disponível: true; senão false (env CI-aware) |

## Riscos

| Risco | Mitigação |
|---|---|
| GNOME 40+ não mostra system tray sem extension `AppIndicator and KStatusNotifierItem Support` | Documentado em `docs/os-integration/tray-setup.md` |
| Wayland + compositors raw (Sway pure, raw Wayland) não suportam StatusNotifierItem | Fail-open documentado |
| `fyne.io/systray` requer main thread no Linux mas Wails também — depende de `RunWithExternalLoop` API funcionar com Wails ativo | **Threading conhecido OK no Linux:** `nativeStart` (em `systray_unix.go:190`) só faz D-Bus + `notifier.ExportStatusNotifierItem`; **nenhuma chamada GTK**. `godbus` é goroutine-safe (mutex protegido). Logo `start()` rodando numa goroutine separada da que Wails usa pra GTK é seguro nesse alvo (macOS e Windows fariam GTK/Win32 calls inline — mas esses não são alvo). Validar na Stage 1 do plano via smoke build + run em DE real. **Escalation plan se quebrar:** (a) tentar `systray.Register` + delegar event loop pro Wails GTK runner; se isso requer fork da lib, (b) ship spec #1 SEM tray (mantém só single-instance + close-to-tray via `HideWindowOnClose=false` forçado). Spec #2 (drag-folder) não depende de tray, segue. Tray volta como tarefa separada |
| `HideWindowOnClose` + tray failure → zumbi | `DetectTray` antes de aplicar; `onExit` fallback dispara `runtime.Quit` |
| `SingleInstanceLock.UniqueId` colide com app de terceiros | Namespace explícito `com.marcosdid.jarvis` |
| Dev tem 2 jarvis no PATH com mesma UniqueId | Mesma UniqueId → 2ª chama focus da 1ª. **É a feature** |
| `runtime.WindowShow` em janela já visível | No-op (Wails handles) |
| Quit acontece mid-write do master Claude | `OnShutdown` chain existente já lida (`masterSvc.Stop` antes do `localSrv.Stop`); spec não muda essa ordem |
