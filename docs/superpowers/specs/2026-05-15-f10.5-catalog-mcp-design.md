# F10.5 — Catalog + MCP server (design)

Status: aprovado para implementação após review.

Esta é a **primeira de duas specs** que cobrem o bloco F10.5 do
[pivote Go+Wails](2026-05-12-pivot-go-wails-native-design.md). A divisão foi
escolhida pelo usuário no kickoff de brainstorm:

| Spec | Conteúdo | Razão |
|---|---|---|
| **#1 — Catalog + MCP** (este doc) | `internal/catalog`, `internal/mcp`, refactor de `internal/hooks` para listener compartilhado | Dois pacotes greenfield acoplados (MCP `create_task` resolve template via catálogo). Risco do SDK MCP isolado aqui. |
| #2 — Master session upgrade | Ai-jail wrap da master session existente, `.claude/settings.json` writer com config MCP + token, `--resume` persistência | Constrói sobre código existente (`internal/master`, `internal/api/master.go`, `MasterSidebar.tsx`). Depende da spec #1 (consome o token bearer do MCP). |

A spec #1 não toca a UI. O dropdown de template no formulário de criar
task é alteração trivial — sai junto com a PR final desta spec.

---

## 1. Contexto

F10.0–F10.4 entregaram: skeleton Wails, store + migrations SQLite, slice
vertical de Tasks (kanban), Projects + Worktrees, e Sandbox + Sessions +
Hooks. F10.5 fecha o conjunto que falta pra **paridade funcional** com o
J-arvis Python: master session privilegiada, catálogo de templates +
permission-profiles, e servidor MCP que expõe operações sobre tasks para
o master Claude consumir como ferramentas.

Esta spec ataca os dois sub-sistemas backend mais acoplados:

- **Catálogo** — templates (presets de fluxo) + permission profiles
  (presets de CLI flags do claude). Hoje em Python: YAML carregado uma
  vez no boot, validado, nunca recarregado, sem CRUD. Quando uma task é
  criada com `template=X`, o profile default do template é "congelado"
  na linha da task; no spawn, o profile é lido do catálogo atual e seus
  `claude_args` aplicados à subprocesso claude.
- **MCP server** — servidor JSON-RPC 2.0 montado em HTTP que expõe 7
  ferramentas (`list_projects`, `get_project`, `list_tasks`, `get_task`,
  `create_task`, `update_task`, `discard_task`). Consumido pela master
  session (claude, com `mcpServers` configurado em `.claude/settings.json`).

### 1.1 Estado atual do código (verificado)

- `internal/master/session.go` + `internal/api/master.go` + UI sidebar
  **já existem** (F10.0-era). Spawnam `claude` em PTY via `creack/pty`,
  emitem `master.output`/`master.status`/`master.exit`. **Sem ai-jail,
  sem settings.json, sem MCP, sem `--resume`.** Spec #2 cuida disso.
- Não existem `internal/catalog`, `internal/mcp`, nem `internal/localhttp`.
- `internal/hooks` tem o seu próprio listener `net.Listener` em
  `127.0.0.1:0` — vai ser refactorado nesta spec.
- `go.mod`: `github.com/creack/pty` já entra indireto via Wails. Vai
  precisar adicionar **duas** deps diretas: `github.com/modelcontextprotocol/go-sdk`
  v1.5.0 e `gopkg.in/yaml.v3`.

### 1.2 Risco #1 resolvido

O pivot spec lista risco #1 como "MCP SDK Go pode não existir ou estar
imaturo, plano B: JSON-RPC 2.0 manual". Validado em
2026-05-15: existe **SDK oficial** `github.com/modelcontextprotocol/go-sdk`
mantido pela Anthropic em colaboração com Google, na release estável
**v1.5.0** (2026-04-07) com **garantia formal de compatibilidade de API**.
Plano B (hand-roll) descartado. Risco #1 retirado.

## 2. Decisões de design (resumo)

| # | Decisão | Razão |
|---|---|---|
| 1 | Duas specs (Catalog+MCP, depois Master) | Spec única ficaria longa, master só começa depois que MCP exporta token bearer. |
| 2 | SDK MCP oficial v1.5.0 (não hand-roll) | Mantido pela Anthropic+Google, garantia de compat, 1:1 com Python que usa SDK oficial Python. |
| 3 | Listener HTTP compartilhado (`internal/localhttp`) | Hooks e MCP no mesmo `127.0.0.1:<port>`. Um lifecycle, um logging, sem refactor caso adicione mais rotas (Run from Panel?). |
| 4 | Catálogo embedded via `go:embed` | É config de app, não dado de usuário. Nunca recarregado em runtime. Editar = recompilar (mesmo workflow do Python: "commit-and-restart"). |
| 5 | MCP tools como sibling delivery adapter de `core` | `api.*API` (Wails) e `mcp.Server` (HTTP) são adapters irmãos sobre `core.*Service`. Eventos sobem grátis pra UI. Aplicação ortodoxa de arquitetura hexagonal. |
| 6 | Bearer token gerado no boot, vive em memória | Single user, loopback only. Token novo a cada boot do daemon. Spec #2 lê o valor e escreve no settings.json da master session. |
| 7 | Paridade de tools 1:1 com Python (validada por snapshot) | Risco #7 do pivot spec: schema drift. Snapshot capturado **antes** da implementação Go. |
| 8 | Catalog erros = fail-fast no boot | YAML está embedded; YAML inválido = bug de compilação, não condição de runtime. `log.Fatalf`. |
| 9 | Domain erros via MCP = `isError: true` + texto | Padrão MCP. Cliente (claude) lê texto e age. JSON-RPC error reservado pra erros de protocolo. |
| 10 | Auth ausente nesta spec só pra UI/Wails | UI não precisa autenticar contra MCP. MCP é só pra master-claude. Auth = bearer único de boot. |

## 3. Arquitetura

### 3.1 Layout de pacotes

```
internal/
  catalog/          NEW. Embedded YAML config, validated once at boot.
    catalog.go         Catalog, Template, PermissionProfile structs + Resolve()
    load.go            //go:embed catalog.yml + validação
    catalog.yml        Portado verbatim de orchestrator/config/catalog.yml
    catalog_test.go    Validation table tests + golden YAML round-trip

  localhttp/        NEW. Shared 127.0.0.1:0 listener para hooks + mcp.
    server.go          Start/Stop/BaseURL, expõe Mount(prefix, http.Handler)
    server_test.go

  mcp/              NEW. Sibling delivery adapter sobre core.
    server.go          Wraps mcp.Server do SDK oficial, registra 7 tools,
                       expõe http.Handler para localhttp.
    tools.go           7 tool handlers — adapters finos sobre core.*Service
    auth.go            Bearer-token middleware (single boot token, crypto/rand 256-bit)
    schema.go          JSON schemas dos tools (snapshot-tested)
    tools_test.go      Per-tool + tools/list parity snapshot

  hooks/            REFACTOR. Perde o listener; vira hooks.Handler.
    server.go          Handler implementa http.Handler — registra /api/hooks/*
                       num mux passado pelo localhttp. Start/Stop/BaseURL saem.

  api/              UNCHANGED. core continua single source of truth.
  core/             ALMOST UNCHANGED — TasksService.Create ganha parâmetro
                    templateName; SessionsService.Start ganha lookup do profile
                    no catálogo.
```

### 3.2 Dependency arrows (todas pra dentro)

```
  Wails (main.go)
        |
  api.*API ----+
               |
               v
            core.*Service  <----+
               ^                |
               |                |
         mcp.Server (tools)     |
               ^                |
               |                |
         localhttp.Server <-- hooks.Handler
```

`mcp` e `api` são **siblings** — ambos dependem de `core`, nenhum depende
do outro. `localhttp` é plumbing puro (só conhece `http.Handler`).

### 3.3 Wiring no `main.go` (novas linhas)

```go
catalogRoot := catalog.MustLoad()                              // fail-fast no boot
mcpToken    := mcp.NewBearerToken()                            // crypto/rand 256-bit
mcpHandler  := mcp.NewServer(tasksSvc, projectsSvc, catalogRoot, mcpToken).Handler()
// lazyBus, hookUpdater, tokenRegistry: reusados do wiring atual do F10.4.
hookHandler := hooks.NewHandler(tokenRegistry, lazyBus, hookUpdater)

localSrv := localhttp.New()
_ = localSrv.Mount("/api/hooks/", hookHandler)
_ = localSrv.Mount("/api/mcp",    mcpHandler)
port, err := localSrv.Start()
if err != nil { log.Fatalf("local http: %v", err) }
log.Printf("local http listening on 127.0.0.1:%d", port)
defer func() { _ = localSrv.Stop() }()

// Spec #2 vai consumir: localSrv.BaseURL() + mcpToken.Value() pra escrever
// no .claude/settings.json da master session.
```

Ordem importa (lição do F10.4.20): `Mount` antes de `Start`, sempre.

## 4. Catalog

### 4.1 Modelo de dados (paridade 1:1 com Python)

```go
package catalog

type PermissionProfile struct {
    Name        string   `yaml:"-"           json:"name"`
    Description string   `yaml:"description" json:"description"`
    ClaudeArgs  []string `yaml:"claude_args" json:"claude_args"`
}

type Template struct {
    Name                     string `yaml:"-"                          json:"name"`
    Description              string `yaml:"description"                 json:"description"`
    DefaultPermissionProfile string `yaml:"default_permission_profile"  json:"default_permission_profile"`
    BranchPrefix             string `yaml:"branch_prefix"               json:"branch_prefix"`
}

type Catalog struct {
    Version                   string                       `yaml:"version"                      json:"version"`
    FallbackPermissionProfile string                       `yaml:"fallback_permission_profile"  json:"fallback_permission_profile"`
    PermissionProfiles        map[string]PermissionProfile `yaml:"permission_profiles"          json:"-"`
    Templates                 map[string]Template          `yaml:"templates"                    json:"-"`
}
```

Os campos `Name` (com `yaml:"-"`) são preenchidos após o unmarshal, a
partir da chave do map. Eles existem pra que `Resolve()` possa devolver
um shape auto-contido.

### 4.2 Carga e validação

```go
//go:embed catalog.yml
var embeddedYAML []byte

var ErrCatalog = errors.New("catalog")

// MustLoad lê o YAML embedded, valida, e mata o processo em erro.
// Chamado uma única vez em main.go.
func MustLoad() *Catalog

// Load aceita YAML arbitrário (pra testes).
func Load(raw []byte) (*Catalog, error)
```

Regras de validação (cópia da `@model_validator` Pydantic):

1. `version == "1"` (literal).
2. `fallback_permission_profile` existe em `permission_profiles`.
3. Todo `template.default_permission_profile` existe em `permission_profiles`.
4. `branch_prefix` de cada template casa `^[a-z][a-z0-9-]*/$`.

**Agregação de erros**: `Load` não para no primeiro erro — coleta todos
e devolve uma string única separada por `;`. Editar um catálogo que tem
4 problemas exige ver os 4, não corrigir-rebuildar 4 vezes.

### 4.3 Resolução no spawn

```go
type Resolved struct {
    TemplateName string   // "" se a task não tem template
    ProfileName  string   // sempre preenchido (default do template ou fallback)
    ClaudeArgs   []string // flags pra passar ao claude
    BranchPrefix string   // "" se task sem template
}

var ErrTemplateUnknown = fmt.Errorf("%w: template unknown", ErrCatalog)
var ErrProfileMissing  = fmt.Errorf("%w: permission profile missing", ErrCatalog)

// Resolve aceita um template name (pode ser ""), retorna o que aplicar
// no spawn. Erro = ErrTemplateUnknown se name != "" e não existe.
func (c *Catalog) Resolve(templateName string) (Resolved, error)

// ResolveProfile busca um profile diretamente pelo nome — usado quando
// SessionsService.Start lê o profile name persistido na task.
// Erro = ErrProfileMissing se o profile foi removido entre rebuilds.
func (c *Catalog) ResolveProfile(profileName string) (PermissionProfile, error)
```

### 4.4 Aplicação no domain layer

Mudanças em `internal/core`:

- `core.TasksService.Create` ganha parâmetro `templateName string`
  (pode ser ""). Internamente chama `catalog.Resolve(templateName)`,
  persiste `template` e `permission_profile` na linha da task.
- `core.SessionsService.Start` (existente) ganha lookup: lê
  `task.permission_profile`, chama `catalog.ResolveProfile(...)`, e
  passa `Resolved.ClaudeArgs` pra `sandbox.Runtime.Spawn` em
  `RuntimeSpec.ExtraArgs`.

**Comportamento se o profile sumiu** (entre dois rebuilds da app a task
ficou apontando pra um profile que foi removido do catálogo): erro
hard — `SessionsService.Start` retorna `ErrProfileMissing`, a sessão
não sobe, e a UI mostra a razão. **Não substituir silenciosamente pelo
fallback** — pode mudar o significado de "permissão" da task. Mesma
postura do Python (decision #7 do F7 spec).

**Comportamento se a task foi criada antes do F10.5** (sem
`permission_profile`): `SessionsService.Start` aplica
`fallback_permission_profile`. Não falha (compatibilidade pra trás).

### 4.5 Migrations / schema

A migration `20260101000001_init.sql` já tem `template TEXT` e
`permission_profile TEXT` na tabela `tasks` (linhas 29-30, portado das
alembic 0001-0006 no F10.1, paridade com Python). **Sem migration nova
nesta spec** — só passar a *escrever* os campos no `tasks.Create`.

### 4.6 API binding

`internal/api/catalog.go` — endpoint somente-leitura pro dropdown na UI:

```go
type CatalogAPI struct{ root *catalog.Catalog }

type CatalogView struct {
    Version                   string              `json:"version"`
    FallbackPermissionProfile string              `json:"fallback_permission_profile"`
    PermissionProfiles        []PermissionProfile `json:"permission_profiles"` // map → lista ordenada por Name
    Templates                 []Template          `json:"templates"`            // idem
}

func NewCatalogAPI(root *catalog.Catalog) *CatalogAPI
func (a *CatalogAPI) Get() CatalogView
```

Bindado no Wails (`app.Bind(catalogAPI)`). Sem `Set`/`Update`. UI consome
pra popular o dropdown de template na criação de task.

## 5. MCP server

### 5.1 Construção e dependências

```go
package mcp

import gosdk "github.com/modelcontextprotocol/go-sdk/mcp"

type Server struct {
    tasks    *core.TasksService     // injetado
    projects *core.ProjectsService  // injetado
    catalog  *catalog.Catalog       // injetado (create_task resolve template)
    token    *BearerToken           // gerado no boot
    mcp      *gosdk.Server          // do SDK oficial
}

func NewServer(tasks *core.TasksService, projects *core.ProjectsService,
               cat *catalog.Catalog, tok *BearerToken) *Server

// Handler retorna o http.Handler pra montar no localhttp. Envolve o
// StreamableHTTPHandler do SDK com o middleware de auth (bearer).
func (s *Server) Handler() http.Handler
```

### 5.2 Os 7 tools (paridade 1:1)

| Tool | Input | Saída | Chama em `core` |
|---|---|---|---|
| `list_projects` | `{}` | `[{id, name, path}]` | `projects.List()` |
| `get_project` | `{project_id: string}` | `{id, name, path}` | `projects.Get(id)` |
| `list_tasks` | `{project_id?: string, state?: enum}` | `[{...task shape...}]` | `tasks.List(filter)` |
| `get_task` | `{task_id: string}` | `{...task shape...}` | `tasks.Get(id)` |
| `create_task` | `{project_id, title, description?, template?, branch?}` | task shape | `tasks.Create(...)` — resolve template→profile internamente |
| `update_task` | `{task_id, title?, description?, state?, branch?}` | task shape | `tasks.Patch(id, patch)` — `template` **não** editável |
| `discard_task` | `{task_id}` | task com `state="discarded"` | `tasks.Patch(id, {state: discarded})` |

Task shape (paridade Python):
```json
{
  "id": "...", "project_id": "...", "title": "...", "description": "...",
  "state": "idea", "branch": "...",
  "template": "frontend", "permission_profile": "yolo"
}
```

Valores válidos de `state` (state machine em `internal/core/task.go`,
verificado em `internal/core/task_test.go`):
`idea | ready | in_progress | review | done | discarded`. Transições
inválidas (ex: `done → in_progress`, `discarded → idea`) emitem
`core.ErrInvalidStateTransition` — vide §5.6.

Cada tool é registrado no SDK assim:

```go
gosdk.AddTool(s.mcp, &gosdk.Tool{
    Name:        "create_task",
    Description: "Create a task in a project, optionally from a template",
    InputSchema: createTaskSchema,  // *jsonschema.Schema em schema.go
}, s.createTaskHandler)
```

O SDK gera `tools/list` automaticamente. Saída de cada tool:
`gosdk.CallToolResult{Content: []gosdk.Content{&gosdk.TextContent{Text: jsonString}}}`
— mesmo formato que o Python.

### 5.3 Bearer token — lifecycle

```go
type BearerToken struct{ value string }

func NewBearerToken() *BearerToken             // crypto/rand 32B → base64url
func (t *BearerToken) Value() string           // pra spec #2 escrever no settings.json
func (t *BearerToken) Validate(s string) bool  // subtle.ConstantTimeCompare
```

- **Gerado uma vez no boot** em `main.go`.
- **Vive em memória** — não persistido, regenera todo restart.
- **Spec #1 só gera e valida**. Spec #2 lê o valor e escreve no
  `.claude/settings.json` da master session.

### 5.4 Auth middleware

```go
// auth.go
func (s *Server) Handler() http.Handler {
    sdkHandler := gosdk.NewStreamableHTTPHandler(s.mcp, ...)
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        hdr := r.Header.Get("Authorization")
        if !strings.HasPrefix(hdr, "Bearer ") {
            http.Error(w, `{"error":"missing bearer token"}`, http.StatusUnauthorized)
            return
        }
        if !s.token.Validate(strings.TrimPrefix(hdr, "Bearer ")) {
            // Log warn (debug ajuda) mas mensagem genérica pro cliente
            log.Printf("mcp: invalid bearer token from %s", r.RemoteAddr)
            http.Error(w, `{"error":"invalid bearer token"}`, http.StatusUnauthorized)
            return
        }
        sdkHandler.ServeHTTP(w, r)
    })
}
```

Sem rate limit (loopback, single user). Constant-time compare obrigatório.

### 5.5 Transporte HTTP

`StreamableHTTPHandler` do SDK encapsula JSON-RPC + session ID + SSE
upgrades. Montado em `/api/mcp` no `localhttp.Server`. Não precisamos
mexer com framing manualmente.

### 5.6 Mapeamento de erros

| Cenário | HTTP | MCP retorna |
|---|---|---|
| Auth ausente/inválido | 401 | (não chega no tool) |
| Tool inexistente | 200 | `code: -32601` (SDK) |
| Validação de input | 200 | `code: -32602` (SDK) |
| `core.ErrTaskNotFound` | 200 | `isError: true`, `"task not found: <id>"` |
| `core.ErrProjectNotFound` | 200 | `isError: true`, `"project not found: <id>"` |
| `core.ErrInvalidStateTransition` | 200 | `isError: true`, `"invalid state transition: <from> -> <to>"` |
| `catalog.ErrTemplateUnknown` | 200 | `isError: true`, `"unknown template: <name>"` |
| panic / DB error | 200 | `isError: true`, `"internal error (see daemon logs)"` |

Erros de domínio = `isError: true` (padrão MCP, cliente lê texto e
decide). JSON-RPC error reservado pra erros de protocolo (auth, schema,
método inexistente).

### 5.7 Eventos

Tools de escrita (`create_task`, `update_task`, `discard_task`) **não
emitem eventos diretamente**. Eles chamam `core.TasksService.*`, que já
emite `task.created`/`task.updated`/`task.discarded` no `events.Emitter`.
O bus é o mesmo da UI Wails — quando master-claude chama um tool, a UI
atualiza em tempo real. Zero código duplicado de fanout.

## 6. Shared internal HTTP server (`internal/localhttp`)

### 6.1 Por que existe

`internal/hooks` hoje faz tudo: cria `net.Listener` em `127.0.0.1:0`,
monta `http.ServeMux`, sobe goroutine, expõe `BaseURL()`. F10.5 precisa
rodar **dois** sub-sistemas no mesmo listener. Lifto o listener pra um
pacote neutro em vez de duplicar a lógica em `mcp/`.

### 6.2 Shape

```go
package localhttp

type Server struct {
    mu       sync.Mutex
    mux      *http.ServeMux
    srv      *http.Server
    listener net.Listener
    started  bool
}

func New() *Server

// Mount registra um http.Handler num prefixo. http.ServeMux trata prefix
// matching (sufixo "/" = subtree match). Chamar Mount após Start = erro.
func (s *Server) Mount(pattern string, h http.Handler) error

func (s *Server) Start() (port int, err error)
func (s *Server) Stop() error                // graceful shutdown 2s timeout
func (s *Server) BaseURL() string            // "http://127.0.0.1:<port>"
func (s *Server) Started() bool              // true após Start bem-sucedido (lido por main.go pra sandbox_available)
```

Invariantes:

- `Mount` antes de `Start` (proteção contra a data race que pegamos no F10.4.20).
- `Start` retorna erro na segunda chamada (não panic).
- `Stop` é safe de chamar 2x (segunda é no-op).
- `Stop` antes de `Start` = no-op.

### 6.3 Refactor do `internal/hooks`

`hooks.Server` perde listener e vira `hooks.Handler`:

```go
// antes:
type Server struct { /* listener + srv + mux + registry + bus + updater */ }
func NewServer(reg *TokenRegistry, bus events.Emitter, u SessionUpdater) *Server
func (s *Server) Start() (int, error)
func (s *Server) Stop() error
func (s *Server) BaseURL() string

// depois:
type Handler struct { /* registry + bus + updater + http.ServeMux interno */ }
func NewHandler(reg *TokenRegistry, bus events.Emitter, u SessionUpdater) *Handler
func (h *Handler) ServeHTTP(w http.ResponseWriter, r *http.Request)
```

Afetados:

- `main.go` — wiring novo (3.3 acima).
- `cmd/jarvis-e2e-http/main.go` — mesmo refactor.
- `internal/api/http_e2e.go` — `SetHookBase(string)` continua igual; a
  string vem agora de `localSrv.BaseURL()`.
- Testes de hooks — usam `httptest.NewServer(handler)` em vez de
  `hooks.Server.Start()`.

`hooks.Server.Started()` (`server.go:81`, lido por `main.go` pra computar
`sandbox_available`) sai. A nova fonte é `localSrv.Started()` (mesmo
contrato — `true` após `Start` bem-sucedido).

### 6.4 Roteamento

```go
mux.Handle("/api/hooks/", hookHandler)   // subtree: /api/hooks/{event}/{token}
mux.Handle("/api/mcp",    mcpHandler)    // exact: /api/mcp (POST + SSE)
```

`net/http.ServeMux` resolve conflitos via specificity. Não precisa de
gorilla/mux ou chi.

### 6.5 O que **não** vai pro `localhttp`

- Auth (cada sub-handler cuida — hooks usa token-na-URL,
  mcp usa bearer header).
- Logging de requests (cada sub-handler loga o que precisa).
- TLS, CORS, rate limit — loopback, single user.

Pacote total: ~80 LOC.

## 7. Tratamento de erros e edge cases

### 7.1 Catalog — fail-fast no boot

`MustLoad` agrega todos os erros e chama `log.Fatalf`. YAML está
embedded; YAML inválido = bug de compilação, não condição de runtime.

Casos cobertos:

- YAML mal-formado → `"catalog: unmarshal: line 5: ..."`
- `version != "1"` → `"catalog: unsupported version \"2\""`
- Fallback profile não existe → `"catalog: fallback_permission_profile \"foo\" not in permission_profiles"`
- Template referencia profile inexistente → `"catalog: template \"frontend\" references unknown profile \"bar\""`
- `branch_prefix` não casa regex → `"catalog: template \"x\": branch_prefix \"Foo/\" must match ^[a-z][a-z0-9-]*/$"`
- Múltiplos problemas → mensagem única, separados por `;`.

### 7.2 Catalog — erros de runtime

Cobertos na seção 4.4 (ErrTemplateUnknown e ErrProfileMissing).

### 7.3 MCP — auth edge cases

- Header ausente → 401 `{"error":"missing bearer token"}`.
- Header presente, valor errado → 401 `{"error":"invalid bearer token"}`.
- Comparação `subtle.ConstantTimeCompare` (nunca `==`).
- Log warn de tentativa falha (não rate-limit, mas dá pra debugar).

### 7.4 `localhttp` edge cases

Cobertos em 6.2 (Mount after Start, Start 2x, Stop sem Start, Stop com
request em voo).

### 7.5 Compatibility — task criada antes de F10.5

Tasks pré-F10.5 não têm `permission_profile`. `SessionsService.Start`
detecta `NULL`/empty e aplica `fallback_permission_profile`. Não falha.

## 8. Testes

### 8.1 Pirâmide (alinhada ao §9.1 do pivot spec)

| Camada | Cobertura |
|---|---|
| **Unit — `internal/catalog`** | Table tests pra cada cenário do 7.1; round-trip YAML; `Resolve()` nos 4 ramos (template existe / não existe / vazio→fallback / profile sumido). |
| **Unit — `internal/mcp`** | Cada tool com `core.*Service` mockado via interface pequena. Bearer auth middleware. Mapeamento de erros do 5.6. |

### 8.2 E2E Playwright

**Pular nesta spec.** A UI só ganha um dropdown de template no formulário
de criar task — alteração trivial. Cobertura mínima fica no test unitário
do componente React (já existe pra `CreateTaskForm`, adicionar caso).

### 8.3 Subprocess-forking

**Nenhum.** Esta spec não forka subprocess. Não precisa de
`//go:build integration` em lugar nenhum. Todos os testes rodam no
caminho normal de `go test ./...`.

### 8.4 Mocks vs reais

Seguindo o feedback memory "integration tests não devem mockar DB":

- Catalog unit: sem DB, lógica pura.
- MCP unit: `core.*Service` mockado via interface pequena (`type tasksReader interface { Get(ctx, id) ... }`), porque a unidade é a tool.
- MCP integration: `core` real + DB real (modernc SQLite in-memory). Sem mock.

- `internal/catalog` ≥ 85%
- `internal/mcp` ≥ 80%
- `internal/localhttp` ≥ 80%
