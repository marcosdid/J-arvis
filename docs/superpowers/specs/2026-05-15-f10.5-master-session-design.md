# F10.5 spec #2 — Master Session Upgrade (design)

Status: aprovado para implementação após review.

Esta é a **segunda de duas specs** do bloco F10.5 do
[pivote Go+Wails](2026-05-12-pivot-go-wails-native-design.md). A primeira
(Catalog + MCP) foi entregue em 2026-05-15 na branch `feat/f10.5-catalog-mcp`
e expôs `mcpSrv.Token()` + `localSrv.BaseURL()` — esta spec consome ambos.

## 1. Contexto

A master session **já existe** no Go pós-F10.0: `internal/master/session.go`
spawna `claude` num PTY via `creack/pty`, `internal/api/master.go` é o
`MasterAPI` ligado ao Wails, e `ui/src/components/master/MasterSidebar.tsx`
renderiza o terminal via `xterm.js`. **Mas** está em modo "F0": claude bare,
sem ai-jail, sem `.claude/settings.json`, sem `--resume` persistente, sem
DB tracking.

O que falta pra paridade com o J-arvis Python (`orchestrator/sandbox/pty_runtime.py`
+ `master_settings_writer.py`):

| Aspecto | Python | Go atual | Esta spec |
|---|---|---|---|
| Binário spawnado | `ai-jail` (com claude dentro) | `claude` direto | Trocar pra `ai-jail` |
| `.claude/settings.json` | bloco `mcpServers` apontando pro `/api/mcp` interno | não escreve | Implementar writer |
| `.ai-jail` config | `command` + `allow_tcp_ports=[mcp_port]` | não escreve | Estender writer existente |
| Persistência | singleton DB row com `claude_session_id, pid` | tabela existe, nenhum código | Implementar repo + service |
| `--resume <uuid>` | sempre passado (uuid novo no 1º boot, mesmo nos seguintes) | nunca passado | Implementar |
| Watchdog | 2s; se subprocess morre antes, re-spawna com uuid novo | nenhum | Implementar |
| Single-instance | DB CheckConstraint (`id='singleton'`) | Migration tem; código não usa | Usar |
| UI session_id | (lê do DB) | hardcoded `'master_001'` | Real session_id do DB |
| Transcript scroll-back | NÃO expõe (xterm.js raw PTY) | mesmo | mantém — fora de escopo |

Tabela `master_session` já está em
`internal/store/migrations/20260101000001_init.sql:88-95` — não precisa
migration nova.

## 2. Decisões de design

| # | Decisão | Razão |
|---|---|---|
| 1 | Paridade total com Python (ai-jail + watchdog + DB) | Escolha explícita do usuário. Mantém comportamento estável que já funcionava. Process isolation do ai-jail vale o custo apesar do `--dangerously-skip-permissions`. |
| 2 | Novo `core.MasterService` (não engrossar `api.MasterAPI`) | Consistente com `core.TasksService`/`SessionsService` (siblings sobre `core`). Singleton + watchdog + DB são domínio. |
| 3 | `master.Session` continua plumbing PTY puro | Já é minimalista. A diferença vs task sessions (PTY-attached vs detached) é estrutural — não tentar reaproveitar `sandbox.Runtime`. |
| 4 | `WriteMasterSettings` separado de `WriteSettings` | Master settings é `mcpServers`-only; task settings é `hooks`-only. Misturar viola SRP, complica os tests, e os shapes JSON são incompatíveis. |
| 5 | Estender `WriteAijailConfig` (não criar 2º writer) | A única diferença é `allow_tcp_ports`. Param `[]int` opcional (nil = vazio). Backward-compat com callers atuais (task sessions passam nil). |
| 6 | Auto-start no boot do app (sem botão "Start master") | Comportamento atual mantido. Se falhar, UI mostra erro, mas o app abre normal. |
| 7 | Watchdog 2s + auto re-spawn com UUID novo | Mesma heurística do Python. Cobre o caso de `--resume` falhar por transcript corrompido. UI re-dispatcha `Start()` ao receber `master.exit{early_exit: true}`. |
| 8 | `Stop` preserva `claude_session_id` (clear `pid` apenas) | Próximo `Start` faz `--resume` no mesmo session. Pra começar do zero, usuário deleta a row manualmente (caso raro). |
| 9 | Master CWD hardcoded em `~/.local/share/j-arvis/master` | Matches Python (`Path.home() / ".local" / "share" / "j-arvis" / "master"`). Não env-overridable nesta spec — YAGNI. |
| 10 | Sem transcript reading do master | Paridade com Python. xterm.js já mostra output ao vivo. Scroll-back full = futura adição barata. |
| 11 | Sem migration nova | Tabela `master_session` já existe (F10.1). |
| 12 | Nenhuma dep nova | Tudo o que precisa já está em `go.mod`. |

## 3. Arquitetura

### 3.1 Layout de pacotes

```
internal/
  master/                        EXISTENTE — fica como plumbing PTY puro
    session.go                     Session{cmd, tty, OnOutput, OnExit, ...}
                                   ADIÇÃO: StartAijail(cwd string) (pid int, err error)

  core/
    master.go                    NOVO — orquestração + DB + watchdog
      MasterService struct {
        mu         sync.Mutex                 // serializa Start/Stop entre Wails e auto-recovery do UI
        repo       *store.MasterSessionRepo
        session    master.Session             // singleton PTY wrapper
        mcpURL     func() string              // injetado como closure (lazy) — resolve só quando Start roda,
                                              // garantindo que localSrv já foi Start()-ado
        mcpToken   *mcp.BearerToken
        masterCwd  string                     // ~/.local/share/j-arvis/master
        bus        events.Emitter
      }
      Start(ctx) (*MasterSession, error)   // singleton + idempotente
      Stop(ctx) error                       // SIGTERM + ClearPID; preserva row
      Status(ctx) MasterStatus              // lê DB + verifica PID vivo
      Send(data string) error
      Resize(rows, cols uint16) error
    master_test.go                 unit tests + watchdog tests

  sandbox/
    settings_master.go           NOVO — WriteMasterSettings(cwd, mcpURL, token)
    settings_master_test.go
    aijail_config.go             MOD — adicionar `allowTCPPorts []int` param

  store/
    master_session.go            NOVO — MasterSessionRepo
      Get(ctx) (*MasterSession, error)                    // ErrMasterSessionNotFound se vazio
      Upsert(ctx, claudeSessionID string, pid int) error  // INSERT OR REPLACE
      ClearPID(ctx) error                                  // UPDATE ... SET pid=NULL
      Delete(ctx) error                                    // DELETE WHERE id='singleton'; idempotente
    master_session_test.go

  api/
    master.go                    MOD — shell sobre core.MasterService
      Antes:  NewMasterAPI(bus, factory, binPath)
      Depois: NewMasterAPI(svc *core.MasterService)
```

### 3.2 Dependency arrows

```
  Wails (main.go)
        |
  api.MasterAPI ──delegates──▶  core.MasterService
                                        |
                                        ├──▶ store.MasterSessionRepo (singleton row)
                                        ├──▶ master.Session (PTY plumbing)
                                        ├──▶ sandbox.WriteMasterSettings
                                        ├──▶ sandbox.WriteAijailConfig (extended)
                                        └──▶ events.Emitter

  ↑ todos pointing inward; nenhum pacote depende de api/
```

### 3.3 Wiring novo em `main.go`

```go
// já existe pós-spec #1:
mcpToken := mcp.NewBearerToken()
mcpSrv := mcp.NewServer(tasksSvc, projectsSvc, catalogRoot, mcpToken)
localSrv.Mount("/api/mcp", mcpSrv.Handler())
// ... localSrv.Start()

// novo nesta spec:
masterRepo := store.NewMasterSessionRepo(db)
masterSvc := core.NewMasterService(
    masterRepo,
    master.New(),                       // master.Session singleton
    func() string { return localSrv.BaseURL() },  // closure — lazy resolution
    mcpToken,                           // <- consome o "_ = mcpToken" placeholder
    masterCwdDefault(),                 // ~/.local/share/j-arvis/master
    lazyBus,
)
masterAPI := api.NewMasterAPI(masterSvc)
```

E na linha de `OnShutdown` do Wails: adicionar `_ = masterSvc.Stop(c)`
antes do `localSrv.Stop()` (sessão precisa parar antes do listener morrer
pra hooks finais funcionarem).

A closure `func() string { return localSrv.BaseURL() }` resolve a URL **só
quando `MasterService.Start` é invocado** — não no momento da construção.
Isso destrava a ordem `core.NewMasterService → localSrv.Mount → localSrv.Start
→ MasterAPI.Start` sem precisar mover o wiring inteiro pra dentro do
`OnStartup` do Wails.

`JARVIS_CLAUDE_BIN` env var sai. Master sempre roda via ai-jail.

## 4. Lifecycle, estados e fluxos

### 4.1 Estados

```
                  ┌──────────────────────────────────┐
                  │                                  │
                  ▼                                  │
    [no row]  ──Start──▶  [running]  ──Stop──▶  [stopped]
                              │                      │
                              │ (subprocess          │
                              │  morre sozinho)      │
                              ▼                      │
                          [crashed]  ───────re-Start─┘
                              │
                              ▼
                          watchdog: se exit ≤ 2s
                          assume --resume falhou
                          → re-spawn com UUID novo
```

Reading the DB row:
- **No row + no PID** → never started; `Start` cria UUID novo + persiste.
- **Row + PID != nil + processo vivo** → already running; `Start` é no-op.
- **Row + PID != nil + processo morto** → crashed; `Start` re-resume com o
  mesmo `claude_session_id`.
- **Row + PID == nil** → stopped cleanly via `MasterService.Stop`; `Start`
  re-resume com o mesmo `claude_session_id`.

**Idempotência e concorrência:** `Start` adquire `s.mu` (mutex no
`MasterService`) durante toda a sequência. Isso serializa chamadas
concorrentes do Wails e do auto-recovery do UI (§6.1). Re-entrância do
`master.exit` handler é segura — o segundo `Start` espera o lock, lê o
DB e detecta o estado pós-recovery.

**Idempotência dos config writers:** `WriteMasterSettings` e
`WriteAijailConfig` ambos sobrescrevem qualquer arquivo prévio. Se step
5 (settings) sucede mas step 6 (.ai-jail) falha, o CWD fica
half-configured — mas isso é self-healing: a próxima invocação de
`Start` sobrescreve o `settings.json` com a URL/token atuais e re-tenta
o `.ai-jail`. Sem rollback explícito.

### 4.2 Fluxo de `MasterService.Start(ctx)`

```go
func (s *MasterService) Start(ctx context.Context) (*MasterSession, error) {
    s.mu.Lock()
    defer s.mu.Unlock()

    // 1. Preflight: ai-jail disponível?
    if err := sandbox.SandboxAvailable(); err != nil {
        return nil, fmt.Errorf("%w: %v", ErrMasterSandboxUnavailable, err)
    }
    // 1b. Preflight: MCP URL conhecida?
    mcpURL := s.mcpURL()
    if mcpURL == "" {
        return nil, ErrMasterMCPUnavailable
    }

    // 2. DB lookup
    row, err := s.repo.Get(ctx)
    if err == nil && row.PID != nil && processAlive(*row.PID) {
        return row, nil  // idempotente
    }

    // 3. Determinar session_id
    sessionID := uuid.NewString()
    if err == nil && row.ClaudeSessionID != "" {
        sessionID = row.ClaudeSessionID
    }

    // 4. Garantir CWD
    if err := os.MkdirAll(s.masterCwd, 0o755); err != nil {
        return nil, fmt.Errorf("create master cwd: %w", err)
    }

    // 5. Configs (ambos sobrescrevem; partial failure é self-healing — ver §4.1)
    if err := sandbox.WriteMasterSettings(s.masterCwd, mcpURL, s.mcpToken.Value()); err != nil {
        return nil, fmt.Errorf("write master settings: %w", err)
    }
    claudeArgs := []string{"--dangerously-skip-permissions", "--resume", sessionID}
    mcpPort := portFromURL(mcpURL)
    if err := sandbox.WriteAijailConfig(s.masterCwd, claudeArgs, []int{mcpPort}); err != nil {
        return nil, fmt.Errorf("write master .ai-jail: %w", err)
    }

    // 6. Spawn PTY
    pid, err := s.session.StartAijail(s.masterCwd)
    if err != nil {
        return nil, fmt.Errorf("spawn master: %w", err)
    }

    // 7. Persistir
    if err := s.repo.Upsert(ctx, sessionID, pid); err != nil {
        _ = syscall.Kill(pid, syscall.SIGTERM)
        return nil, fmt.Errorf("persist master row: %w", err)
    }

    // 8. Watchdog
    go s.watchdog(sessionID, pid)

    s.bus.Emit("master.status", MasterStatus{Running: true, PID: pid, SessionID: sessionID})
    return &MasterSession{ClaudeSessionID: sessionID, PID: &pid, StartedAt: time.Now()}, nil
}
```

### 4.3 Fluxo de `Stop(ctx)`

```go
func (s *MasterService) Stop(ctx context.Context) error {
    row, err := s.repo.Get(ctx)
    if err != nil {
        return nil  // nada pra parar
    }
    if row.PID != nil {
        _ = syscall.Kill(*row.PID, syscall.SIGTERM)
        // watchdog limpa o pid no DB quando notar o exit
    }
    if err := s.repo.ClearPID(ctx); err != nil {
        return err
    }
    s.bus.Emit("master.status", MasterStatus{Running: false, SessionID: row.ClaudeSessionID})
    return nil
}
```

### 4.4 Watchdog

```go
func (s *MasterService) watchdog(sessionID string, pid int) {
    start := time.Now()
    waitForProcessExit(pid)  // polling syscall.Kill(pid, 0) cada 200ms até ESRCH
    elapsed := time.Since(start)

    early := elapsed < 2*time.Second

    if early {
        // --resume provavelmente falhou (transcript corrompido, ID stale, etc.).
        // Apaga A ROW INTEIRA. O próximo Start verá "no row" e gerará UUID novo.
        // NÃO gravamos um UUID novo aqui porque ele NÃO TEM transcript em disco;
        // gravar levaria a um loop de --resume contra UUIDs fantasmas.
        log.Printf("master watchdog: exit after %v; deleting row for fresh start", elapsed)
        _ = s.repo.Delete(context.Background())
    } else {
        // Saída normal: preserva session_id pra próxima --resume; só limpa PID.
        _ = s.repo.ClearPID(context.Background())
    }

    s.bus.Emit("master.exit", map[string]any{
        "session_id": sessionID,
        "early_exit": early,
        "elapsed_ms": elapsed.Milliseconds(),
    })
}
```

**Mudança vs revisão anterior:** o watchdog não pré-grava `uuid.NewString()`
no DB. No early-exit ele **deleta a row inteira**. Isso garante que o
próximo `Start` (disparado pelo handler do UI ou pelo próximo boot do app)
cai no caminho "**No row + no PID** → gera UUID novo" do §4.1 — sem
nenhum `--resume` contra um UUID fantasma sem transcript em disco.

`MasterSessionRepo` ganha um método `Delete(ctx) error` (DELETE WHERE
id='singleton'; idempotente — sem erro se row não existir).

UI recebe `early_exit: true` e dispara `MasterAPI.Start()` automaticamente
(que vê DB vazio e spawna com UUID novo gerado dentro do próprio Start).

`processAlive(pid)`: `syscall.Kill(pid, 0) == nil`. Não é portable além de
Unix, mas o app inteiro é Linux-only (Wails+WebKitGTK).

## 5. Config writers

### 5.1 `WriteMasterSettings(cwd, mcpBaseURL, bearerToken string) error`

Escreve `<cwd>/.claude/settings.json` com:

```json
{
  "mcpServers": {
    "j-arvis-master": {
      "type": "http",
      "url": "http://127.0.0.1:<port>/api/mcp",
      "headers": {
        "Authorization": "Bearer <token>"
      }
    }
  }
}
```

- File mode `0o600` (owner-only — contém bearer token).
- `MkdirAll(<cwd>/.claude, 0o755)` antes de escrever.
- **Sem bloco `hooks`** — master não usa hook callbacks (status visual vem direto do PTY).

### 5.2 `WriteAijailConfig(cwd string, claudeArgs []string, allowTCPPorts []int) error`

Assinatura existente: `WriteAijailConfig(cwd, claudeArgs)`. Nova: adiciona
`allowTCPPorts []int`.

Para o master, `allowTCPPorts = [<mcp_port>]`. Para task sessions
(`core/sessions.go`), continuar passando `nil`.

**Breaking change — call sites a atualizar:**
- `internal/core/sessions.go` (1 chamada de produção)
- `internal/sandbox/aijail_config_test.go` (6 chamadas de teste — atualizar todas para passar `nil` no novo param)
- Qualquer outro `WriteAijailConfig(` que aparecer no `grep`

Output (formato consumido pelo binário ai-jail):
```
command = ["claude", "--dangerously-skip-permissions", "--resume", "<uuid>"]
allow_tcp_ports = [55821]
rw_maps = [...]
```

`nil` ou `[]int{}` omite a chave inteira (backward-compat com o consumidor
ai-jail, que tem default-deny).

## 6. UI

### 6.1 `MasterSidebar.tsx` mudanças

```tsx
const [sessionId, setSessionId] = useState<string>('');

MasterAPI.Start()
  .then((st) => {
    setSessionId(st.session_id);  // novo campo no MasterStatus
    setStatus(st.running ? 'connected' : 'disconnected');
    setPid(st.running ? st.pid : null);
  });

EventsOn('master.status', (s: unknown) => {
  const obj = s as { running?: boolean; pid?: number; session_id?: string };
  if (obj?.session_id) setSessionId(obj.session_id);
  // ... resto igual
});

EventsOn('master.exit', (payload: unknown) => {
  const obj = payload as { early_exit?: boolean; elapsed_ms?: number };
  if (obj?.early_exit) {
    setSystemMsg({
      level: 'warn',
      message: `master crashed after ${obj.elapsed_ms}ms — recovering...`,
    });
    MasterAPI.Start().then((st) => {
      setSessionId(st.session_id);
      setStatus('connected');
      setPid(st.pid);
      term.write('\x1b[2m-- recovered with new session --\x1b[0m\r\n');
    });
  } else {
    setSystemMsg({ level: 'error', message: `master exited (${obj.elapsed_ms}ms)` });
    setStatus('disconnected');
    setPid(null);
  }
});
```

**Re-entrância de `Start`:** chamar `MasterAPI.Start()` de dentro de um
event handler é seguro porque `MasterService.Start` adquire `s.mu` antes
de tudo (§4.2 step 0) e o lookup do DB em step 2 detecta o estado
pós-recovery (row foi deletada pelo watchdog no early_exit, então a
re-chamada cai no caminho "no row + no PID → gera UUID novo").

### 6.2 `MasterHeader.tsx`

Hoje recebe `sessionId='master_001'` como prop. Agora recebe o
`claude_session_id` real (UUID). Display: trunca pra primeiros 8 chars + `…`
(UUID inteiro é visualmente ruim numa header pequena).

`onCopyId` copia o UUID completo (já funciona).

## 7. Erros e edge cases

### 7.1 Erros tipados (`internal/core/master.go`)

```go
var (
    ErrMasterSandboxUnavailable = errors.New("master: sandbox (ai-jail) unavailable")
    ErrMasterMCPUnavailable     = errors.New("master: MCP URL not yet available (localSrv not started)")
    ErrMasterAlreadyRunning     = errors.New("master: already running")
    ErrMasterSpawnFailed        = errors.New("master: spawn failed")
    ErrMasterDBCorrupt          = errors.New("master: db row in inconsistent state")
)
```

`api.MasterAPI` traduz para mensagens humanas no caminho de erro.

### 7.2 Tabela completa de edge cases

| Cenário | Comportamento |
|---|---|
| `ai-jail` não no PATH | `Start` → `ErrMasterSandboxUnavailable`; UI erro vermelho, sem auto-retry |
| FS perm denied no master CWD | `Start` retorna erro do FS; UI mostra path |
| `pty.Start` falha (PTY exhaustion) | Erro do creack/pty; UI mostra |
| DB PID vivo mas processo morreu | `processAlive=false` → trata como crashed → re-resume |
| DB PID=0 (do crash recovery anterior) | Trata como stopped → re-resume com session atual |
| Crash <2s | watchdog deleta a row inteira (§4.4); UI re-chama `Start`; Start cai em "no row + no PID" → gera UUID novo + spawn fresco |
| Crash >2s (real exit) | `master.exit{early_exit:false}`; UI erro; **sem auto-retry** |
| App fecha com master rodando | `OnShutdown` chama `Stop` (SIGTERM + ClearPID); row preservada |
| App crasha sem `OnShutdown` | PID stale no DB; próxima boot: `processAlive=false` → re-resume |
| MCP server falhou ao subir / `localSrv` não started | Preflight em §4.2 step 1b detecta `mcpURL() == ""` e retorna `ErrMasterMCPUnavailable`; UI mostra erro humano em vez de spawnar claude que tenta conectar em URL vazia |
| Transcript JSONL apagado (corresponde ao `--resume`) | claude crasha → watchdog `early_exit:true` → re-spawn com UUID novo. **É exatamente pra isso que o watchdog existe.** |
| 2 instâncias do app | A 2ª vê PID vivo da 1ª → no-op idempotente. Proteção daemon-level fica pro F10.7 (D-Bus name lock) |

### 8.2 Watchdog test pattern

Watchdog roda em goroutine. Tests usam `done := make(chan struct{})` que
o watchdog fecha ao sair, + `<-done` no test pra evitar leak.

`waitForProcessExit` é abstraído via injection (interface ou function var)
pra que tests possam controlar quando "o processo morreu" sem fork real.
