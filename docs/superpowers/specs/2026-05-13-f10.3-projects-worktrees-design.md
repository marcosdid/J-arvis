# F10.3 — Projects + Worktrees (full port to Go+Wails)

**Date:** 2026-05-13
**Status:** Design (awaiting user review)
**Predecessors:** F10.0/F10.1/F10.2 (Wails skeleton + store + tasks slice + master pty + E2E harness)
**Successor:** F10.4 (Sessions + Sandbox + Hooks) consumes the WorktreesService and adds `git worktree add` via the same `GitOps`.

## 1. Goal

Bring the J-arvis Python "Projects & Worktrees" surface across to Go+Wails so the F9 UI's projects drawer becomes functional in the native app, and so task transitions to terminal state clean up their git worktrees with the same tolerance-on-failure semantics the Python version has.

The slice covers **discover, list, sync, delete, cleanup-on-terminal**. Worktree *creation* (the `git worktree add` flow) is internal to session start and lands with F10.4.

### 2.3. Domain model

A **Project** is a directory on disk pointing at one or more git repositories:

- **Monorepo case:** `basePath/.git/` exists → 1 `Repository` row with `sub_path="."`. Repository name = project name (per Python convention: monorepo borrows the user-given name; sub-repos keep the child dir name).
- **Multi-repo case:** scan immediate children of `basePath`; each child with `<child>/.git/` becomes a `Repository` row with `sub_path=<child name>`. Alphabetical order.
- **Submodule case:** `<child>/.git` as a **file** (gitlink) is skipped — submodules are not independent repos.
- **Neither:** raise `NoGitReposError` (→ 422 from API).

A **Worktree** is a row pointing at a `git worktree`. `(repository_id, path)` is the natural key; `path` is globally unique per schema. `task_id` is nullable:

- `task_id IS NULL` ⇒ **orphan**. Either discovered during sync (user ran `git worktree add` outside the app) or left over from a failed cleanup.
- `task_id` set ⇒ owned by an active task. Cannot be deleted via `DELETE /worktrees/{id}` — must go through `CleanupTaskWorktrees` when the task transitions to terminal state.

## 3. Architectural decisions

### Decision: subprocess `git` CLI, not go-git

`go-git` does not support worktrees (long-standing open issue 1413). The only practical pure-Go option is shelling out, which is what the Python ref does. We get behavioral parity for free and the subprocess interface remains easily mockable. Cost: users must have `git` on PATH — acceptable for a developer tool aimed at engineers.

### Decision: sync-on-list (no fsnotify watcher)

`list_project_worktrees` in Python re-runs `git worktree list --porcelain` every call. We keep that. Rationale: opening the drawer is the user's intent-to-see; `git worktree list` typically runs in <10ms; an fsnotify watcher would complicate the lifecycle for no observed UX gain.

### Decision: cleanup runs **after** the task state-change emit

Sequence in `TasksAPI.Patch` when state → terminal:

1. `repo.UpdateState(ctx, id, state)` — DB commit
2. `bus.Emit("task.updated", task)` — UI invalidates kanban immediately
3. `worktreeCleanup(ctx, id)` — subprocess git, 50–500ms per worktree
4. Cleanup emits `worktree.removed` or `worktree.orphaned` per worktree as it goes

This keeps the kanban transition snappy: state moves visually first, worktrees disappear from the drawer a beat later. If cleanup is reordered before the emit, the drag-to-Done feels frozen.

### Decision: cleanup tolerance — failures orphan, never block

If `git worktree remove --force` fails (filesystem write-protected, mid-commit, IO error), we `UPDATE worktrees SET task_id = NULL` for that row and emit `worktree.orphaned` with the stderr as `reason`. We never propagate cleanup errors back to the task patch. Reason: a task can't get unstuck if it can't transition to Done because of an unrelated worktree issue. Cleanup is best-effort housekeeping; the user can re-try via `DELETE /worktrees/{id}` later.

### Decision: cross-package wiring via function type, not interface

`TasksAPI` takes a `WorktreeCleanupFn func(ctx context.Context, taskID string) error`, not a `WorktreeCleanup interface { CleanupForTask(...) }`. This avoids `api → core → api` cycles or forcing a shared `interfaces` package. The cost (less explicit dependency) is small; the type-alias style keeps `internal/api/tasks.go` from importing anything in `internal/core/worktrees`.

### Decision: emit after commit — but `DeleteOrphan` and `CleanupForTask` have different shapes

Two distinct patterns, do not conflate them:

**`CleanupForTask` (multiple worktrees, partial-failure tolerant):**

1. Load all worktree rows for the task.
2. For each: run `git worktree remove --force`; on failure, mark the row to be orphaned; on success, mark for delete.
3. Apply all DB changes (deletes + UPDATE-task_id-NULL) and accumulate `[]events.EmitCall`.
4. `commit()`
5. Loop the accumulated emits → `bus.Emit`.

If commit fails after some git removes succeeded, the disk is ahead of the DB. Acceptable: next sync will discover the missing rows as already-removed (no harm) and the still-present rows as orphans (correctable). Same trade-off Python accepts.

**`DeleteOrphan` (single worktree, fail-fast):**

1. Load row; check `task_id IS NULL` → 422 if not.
2. `git worktree remove --force` → if errors, **propagate** (this is the user explicitly asking to delete; failure is informative).
3. Delete row.
4. `commit()`
5. Emit `worktree.removed`.

If `git remove` succeeds but `commit` fails, the row lingers but the worktree is gone from disk. Next sync will silently reconcile (the now-missing path won't be discovered, so the row remains in DB pointing at nothing). This is a known acceptable Python quirk; we keep it.

### Decision: `SyncProjectWorktrees` does NOT delete stale DB rows

Python's `list_project_worktrees` upserts discovered entries and returns the union of "in DB" + "in git". Rows whose path no longer appears in `git worktree list` are left alone. Rationale: a task-owned worktree temporarily missing (race with manual `git worktree remove`) should not be silently removed from the DB — that loses task linkage. Stale rows surface as `is_orphan: false` (when task_id is set) even though the worktree no longer exists; the UI shows them as live until `CleanupForTask` runs at terminal transition. This is a known limitation, not a bug — `DeleteOrphan` is the user-driven reconciler for the orphan case.

The taxonomy below matches the Python originals so the existing `ui/src/lib/ws.ts` shim adapts without modification.

| Event name | Payload shape | Fired by |
|---|---|---|
| `worktree.created` | `{project_id, repository_id, worktree_id, path, branch}` | F10.4 only — placeholder in this slice (Sync may emit when a discovered orphan is **new** to the DB; payload `task_id: null`) |
| `worktree.removed` | `{project_id, worktree_id, path, task_id: string|null}` | `DeleteOrphan` (task_id=null) and `CleanupForTask` (task_id=current) — both after commit |

## 7. Cross-cutting: TasksAPI wires WorktreesService.CleanupForTask

**Both `Patch` AND `Discard` get the hook.** Today `TasksAPI.Discard` bypasses `Patch` and calls `repo.Discard` directly; the spec is explicit about adding the cleanup call in **both** paths.

```go
// internal/api/tasks.go
type WorktreeCleanupFn func(ctx context.Context, taskID string) error

type TasksAPI struct {
    repo              TasksRepo
    bus               events.Emitter
    worktreeCleanup   WorktreeCleanupFn  // nil-tolerant: allows unit tests to omit
}

func NewTasksAPI(repo TasksRepo, bus events.Emitter, cleanup WorktreeCleanupFn) *TasksAPI

func (a *TasksAPI) Patch(id string, in PatchTaskInput) (*store.Task, error) {
    // existing logic — fetch current, validate transition, UpdateState, emit task.updated
    // ADDS at the end of the state-changing branch:
    if isTerminal(*in.State) {
        a.runWorktreeCleanup(ctx, id)
    }
    return updated, nil
}

func (a *TasksAPI) Discard(id string) error {
    // existing logic — repo.Discard, fetch full, emit task.discarded
    // ADDS at the end (discard ALWAYS terminal):
    a.runWorktreeCleanup(ctx, id)
    return nil
}

func (a *TasksAPI) runWorktreeCleanup(ctx context.Context, id string) {
    if a.worktreeCleanup == nil { return }  // unit tests can omit
    if err := a.worktreeCleanup(ctx, id); err != nil {
        log.Printf("worktree cleanup for task %s: %v", id, err)  // tolerant — never propagated
    }
}
```

`isTerminal(state string) bool` lives in `internal/core/task.go` next to `IsValidTransition`. Set: `{"done", "discarded"}`.

**Test assertion:** `TasksAPI` unit tests use a fake `WorktreeCleanupFn` and assert `cleanup invoked exactly once on Discard` AND `cleanup invoked exactly once on Patch to done/discarded` AND `cleanup invoked zero times on Patch to non-terminal states`.

Wire-up in `main.go` and `cmd/jarvis-e2e-http/main.go`:

```go
gitOps := git.NewSubprocessOps()
repositoriesRepo := store.NewRepositoriesRepo(db)
worktreesRepo := store.NewWorktreesRepo(db)
worktreeSvc := core.NewWorktreesService(worktreesRepo, repositoriesRepo, projectsRepo, gitOps, lazyBus)
projectsSvc := core.NewProjectsService(projectsRepo, repositoriesRepo, gitOps, lazyBus)

tasksAPI := api.NewTasksAPI(tasksRepo, lazyBus, worktreeSvc.CleanupForTask)
projectsAPI := api.NewProjectsAPI(projectsSvc)         // single-dep: service only
worktreesAPI := api.NewWorktreesAPI(worktreeSvc)        // single-dep: service only
```

**API-layer dependency rule:** API binds depend on `*core.Service`, not directly on `*store.Repo`. The service owns repos and emits events; the API layer is pure HTTP/binding glue. `WorktreesAPI.ListByProject(id)` is `a.svc.SyncProjectWorktrees(ctx, id)` plus shape transform to `WorktreeRead`; `WorktreesAPI.Delete(id)` is `a.svc.DeleteOrphan(ctx, id)`.

Both `main.go` and `cmd/jarvis-e2e-http/main.go` duplicate this. If it gets ugly, extract `func wireDomain(db, bus) (*api.TasksAPI, *api.ProjectsAPI, *api.WorktreesAPI, *api.MasterAPI)` into a small `internal/wire/wire.go` — done during F10.3.10 if needed.

## 8. Error handling

| Source error | Wails error message | HTTP status (for E2E shim) |
|---|---|---|
| `git.NoGitReposError` | `"no .git directory at <path> or 1 level below"` | 422 |
| `store.ErrProjectNotFound` | `"project not found: <id>"` | 404 |
| `core.ProjectHasTasksError` | `"project has <N> task(s); discard them before deleting"` | 409 |
| `git path-unique constraint` (SQLite UNIQUE) | `"a project with that path already exists: <path>"` | 409 |
| `store.ErrWorktreeNotFound` | `"worktree not found: <id>"` | 404 |
| `core.WorktreeNotOrphanError` | `"worktree <id> belongs to active task <task_id>; use task cleanup flow instead"` | 422 |
| `git.GitWorktreeError` | `"git worktree <op> failed in <repo>: <stderr>"` | 500 (only `DeleteOrphan` propagates; `CleanupForTask` swallows) |

Cleanup errors during task transition are **logged at WARN, never returned**.

## 9. Testing strategy

### 9.1. Layer 1 — unit Go, fake `GitOps`

`internal/core/worktrees_test.go` uses a `fakeGit` that records calls and lets each test prime list results / inject removal failures by path. Real SQLite via `newTestDB(t)`. Coverage on `core`:

- `SyncProjectWorktrees`: project not found → 404; discovers new orphans; updates existing branch; multi-repo aggregation.
- `CleanupForTask`: happy path removes everything; one git failure orphans that row + emits `worktree.orphaned`, others still removed; rmdir empty parent dir.
- `DeleteOrphan`: not found; not orphan (task_id set) → 422; git error → 500 (propagated).
- Event ordering: state-changing DB ops happen first; bus.Emit called only after commit.

### 9.2. Layer 2 — integration Go, real `git`

`internal/git/ops_test.go` and `internal/git/detect_test.go` build synthetic git structures in `t.TempDir()`:

```go
shell(t, dir, "git init && git -c user.email=t@t -c user.name=t commit --allow-empty -m init")
```

5–7 tests against `SubprocessOps`: empty list, list with N, add+list shows new, remove drops, remove of non-existent errors, parse handles detached HEAD, parse handles bare worktree. `detect_test.go` covers monorepo, multi-repo (2+ subdirs), gitlink-as-file skipped, empty dir errors.

`internal/store/{repositories,worktrees}_test.go` use the existing `newTestDB(t)` helper. Cover constraint paths (path-unique, project-id FK), upsert semantics, orphan row update.

### 9.3. Layer 3 — Playwright E2E

`ui/e2e/projects-flow.spec.ts` extends the existing harness. New helper at `ui/e2e/helpers.ts`:

```ts
async function makeGitDir(name: string): Promise<string> {
  // tmp/jarvis-e2e-<pid>/projects/<name>/.git/  via fetch to /e2e/fixtures/init-git
}
```

The e2e HTTP server gains a fixture endpoint `POST /e2e/fixtures/init-git` that runs `git init` inside a tmp dir and returns the absolute path. This is the **only** place E2E code touches the host FS through the HTTP backdoor — keeps Playwright tests pure.

Tests (5):

1. **Create monorepo project** — `init-git` fixture → `createProject(name, path)` → expect `project.repositories.length === 1`.
2. **Create multi-repo project** — fixture with 2 sub-`.git`s → `repositories.length === 2`, names sorted alphabetically.
3. **Create rejects no-git path** — fixture returns a path without `.git/` → `createProject` rejects `HTTP 422`.
4. **List worktrees discovers orphan** — after `init-git`, fixture runs `git worktree add` outside the app → `listWorktrees(projectId)` returns 1 entry with `is_orphan: true`.
5. **Discard task with worktree** — seed task + manual worktree row pointing at it via `/e2e/worktrees/__seed` → `tasks.discard(id)` → poll `listWorktrees` until empty → assert `worktree.removed` event was emitted (the e2e shim records emits).

## 10. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| `git worktree list --porcelain` format drift between git versions | Low | Med | Parser test covers detached + bare + branch cases against locally-installed git (which is what users will have). |
| Race: user runs `git worktree remove` externally between sync and cleanup | Med | Low | Cleanup treats "not a working tree" stderr as success — drop the row, no `worktree.orphaned` emit. |
| `project.path` shell escape / path injection | Low | High | `os/exec.Command("git", "-C", path, ...)` uses arg-array form, no shell. Path is validated absolute in `core.CreateProject`. |
| Cleanup deadlock / hang on git operation | Low | Med | `SubprocessOps.Remove` uses `context.WithTimeout(ctx, 30s)` — same as Python. |
| Spec scope creep (worktree create from UI) | High | Med | Explicit out-of-scope in §1. F10.4 owns it. |
| Wire-up duplication between main.go and cmd/jarvis-e2e-http | Med | Low | Acceptable until it grows to 3+ entry points; then extract to `internal/wire/`. |
| Cleanup-during-Patch silently masks real bugs (cleanup never tested via patch path) | Med | Med | TasksAPI tests use a fake `WorktreeCleanupFn` and assert it's invoked exactly once on transition to `done`/`discarded`, zero times otherwise. |

## 11. Out of scope

- **Worktree create via UI** — defer to F10.4 where it's natural (start_session). The internal `GitOps.Add` is implemented and tested now so F10.4 has nothing to add at the adapter layer.
- **Drag-folder-to-create-project from Nautilus** — that's OS integration (F10.7).
- **Path validation beyond `detect_repos`** — no permissions check, no resolved-symlink dedup. Python doesn't do it; we don't either.
- **Concurrent multi-project sync optimization** — sync is per-project, serial. With <10 repos per project and <50ms per `git worktree list`, the perf is fine for single-user.
- **Migration of existing data** — F10.1 starts from an empty DB. There is no "v0.9 Python DB to import."
