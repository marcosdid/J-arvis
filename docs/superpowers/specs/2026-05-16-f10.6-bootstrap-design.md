# F10.6 spec #2 — Manifest Bootstrap (design)

> **Status:** design (brainstorm 2026-05-16). Sibling de F10.6 spec #1 (Run from
> Panel — happy path). Spec #1 trata o caminho "manifest existe, sobe stack";
> spec #2 trata "manifest falta, ofereça gerar".

## Problema

F10.6 spec #1 entregou `RunsService.StartRun` que lê
`<worktree>/.orchestrator/run.yml`. Quando o arquivo não existe, retorna
`ErrManifestMissing` e a UI mostra mensagem genérica "no manifest at ... — commit
one to enable Run". O usuário fica preso: não sabe o schema do YAML nem o que
descrever sobre a stack do projeto.

Spec #2 resolve isso oferecendo um botão **"Iniciar bootstrap"** que abre uma
sessão Claude efêmera dentro do worktree com um único propósito: ler o repo
(README, Dockerfile, package.json, etc.) e propor o `.orchestrator/run.yml`.

## Não-objetivos

- Manifest discovery automático (sem sessão Claude). YAGNI — heurística cheia de
  ifs nunca acerta projetos atípicos. Deixa Claude raciocinar.
- Editor de manifest no UI. YAGNI — Claude já é o editor.
- Histórico/versão de manifests bootstrap. YAGNI.
- Recuperação automática de Claude bootstrap órfão (PID detached após crash da
  app). Documentado como limitação aceita.

## Decisões de design (do brainstorm)

| Pergunta | Resposta |
|---|---|
| Comportamento pós-detecção | **Auto-start com janela de aprovação**: UI mostra preview YAML + countdown 10s + `[Run agora]` `[Cancelar]` |
| Mecanismo de watch | **fsnotify** event-driven (não polling) |
| Modelo de sessão | **`BootstrapService` novo**, usa `sandbox.Runtime` direto, sem tabela SQLite, sem hooks, sem `.claude/settings.json` |
| Como instruir o Claude bootstrap | **`<cwd>/.orchestrator/BOOTSTRAP_PROMPT.md`** embedado via `//go:embed`, escrito antes do spawn |
| Diretório | **Worktree do task** (mesmo `cwd` que `RunsService.DeriveRunCWD`); auto-cria worktree via `WorktreesService.CreateForTask` |
| Timeout | **Sem timeout** — watcher vive enquanto a app rodar; cleanup só por Cancel/CleanupForTask |
| Permission profile | **`fallback_permission_profile`** do catálogo |
| Idempotência | **Retorna sessão existente** quando Start chamado 2x pro mesmo task |
| Cleanup do BOOTSTRAP_PROMPT.md | **Removido automaticamente** quando watcher detecta manifest válido |

## Arquitetura

### Componente novo: `BootstrapService`

Sibling de `RunsService` e `SessionsService` em `internal/core/`. Mantém estado
em memória (sem persistência) — bootstrap é efêmero por design.

```go
type BootstrapService struct {
    runtime  sandbox.Runtime
    wtSvc    *WorktreesService
    tasks    *store.TasksRepo
    catalog  *catalog.Catalog
    bus      events.Emitter

    mu     sync.Mutex
    active map[string]*bootstrapEntry  // keyed by task_id
}

type bootstrapEntry struct {
    sessionID string             // UUID; retornado à UI
    cwd       string             // worktree path
    handle    sandbox.Handle     // PID do Claude bootstrap
    cancel    context.CancelFunc // cancela goroutine do watcher
    startedAt time.Time
}
```

### Os três Claudes em jogo (esclarecimento)

| Claude | Persistência | Hooks | MCP |
|---|---|---|---|
| **Master** (painel lateral, F10.5 spec #2) | Sim, UUID na DB, `--resume` | Sim | Sim (tasks) |
| **Task session** (terminal nativo, F10.4) | Sim, linha em `sessions` | Sim (status semântico) | Não |
| **Bootstrap** (este spec) | **Não** — só `map[task_id]*bootstrapEntry` em RAM | Não | Não |

### Race condition: `handleDetection` vs `Cancel`

Watcher detecta `run.yml` ao mesmo tempo que UI chama Cancel. Quem ganha?
Resolução: single-locked critical section que reivindica ownership atomicamente.

```go
func (b *BootstrapService) handleDetection(taskID, path string) {
    content, err := os.ReadFile(path)
    if err != nil { return }   // race com delete; ignora
    valid, errs := validateManifestContent(content)

    var entry *bootstrapEntry
    b.mu.Lock()
    e, ok := b.active[taskID]
    if !ok {
        b.mu.Unlock()
        return  // Cancel já levou; desiste sem emitir
    }
    if valid {
        delete(b.active, taskID)  // claim
        entry = e
    }
    b.mu.Unlock()

    if entry != nil {
        entry.cancel()
        _ = b.runtime.Kill(context.Background(), entry.handle)
        _ = os.Remove(filepath.Join(entry.cwd, ".orchestrator", "BOOTSTRAP_PROMPT.md"))
    }
    b.bus.Emit("bootstrap.proposed", BootstrapProposedPayload{
        TaskID: taskID, SessionID: e.sessionID,
        ManifestText: string(content), Valid: valid, Errors: errs,
    })
}
```

Garantias:
- `valid=false` → entry permanece, watcher continua, próximo Write dispara novo
  handleDetection. Naturalmente idempotente para eventos duplicados do fsnotify
  (vim/IDE salvando via tmp+rename).
- `valid=true` → exatamente uma das rotas (watcher ou Cancel) faz cleanup; a
  outra vê `!ok` e desiste.

### Cleanup chain (TasksService → terminal state)

Ordem nova: **bootstrap → sessions → runs → worktrees**.

Bootstrap primeiro pelo único motivo objetivo: **é o mais barato de matar**
(sem containers, sem volumes, sem rede). Sessions/runs/worktrees seguem na
ordem atual de `tasks.go:137-139` e `tasks.go:160-162`, que ganham uma chamada
extra `s.runBootstrapCleanup(ctx, id)` no topo. (O argumento "antes do worktree
desaparecer pra evitar órfão de `BOOTSTRAP_PROMPT.md`" vale pra qualquer posição
exceto pós-worktrees — não é o critério decisivo.)

## Contratos

### Camadas Go-internal vs Wails-serialized

**Importante**: o `BootstrapService.Start` retorna um struct **Go-only** com um
canal de sincronização usado pelos testes (`WatcherReady`). A camada
`BootstrapAPI` projeta isso pro JSON serializável que vai à UI via Wails.
Mesma pattern usada em `internal/core/sessions.go` → `internal/api/sessions.go`.

```go
// internal/core/bootstrap.go — retorno do service (Go-only)
type StartedBootstrap struct {
    SessionID     string
    Cwd           string
    ManifestPath  string                  // <cwd>/.orchestrator/run.yml
    PromptPath    string                  // <cwd>/.orchestrator/BOOTSTRAP_PROMPT.md
    WatcherReady  <-chan struct{}         // closed pela goroutine do watcher na 1ª iteração do select; usado por testes pra evitar perder events
}

func (b *BootstrapService) Start(ctx context.Context, taskID string) (*StartedBootstrap, error)
func (b *BootstrapService) Cancel(ctx context.Context, taskID string) error
func (b *BootstrapService) CleanupForTask(ctx context.Context, taskID string) error
```

```go
// internal/api/bootstrap.go — Wails-bound (JSON-serializável)
type BootstrapView struct {
    SessionID    string `json:"session_id"`
    Cwd          string `json:"cwd"`
    ManifestPath string `json:"manifest_path"`
    PromptPath   string `json:"prompt_path"`
}

func (a *BootstrapAPI) Start(taskID string) (BootstrapView, error)  // strips WatcherReady
func (a *BootstrapAPI) Cancel(taskID string) error
```

`SessionID` é debug/log only — a UI hoje **não branca** em cima dele; o modal
chaveia por `task_id`. Mantemos no contrato pra logs e futura observabilidade.

### StartRun result com hint de bootstrap

`RunsAPI.Start` muda de `(RunView, error)` para `(StartRunResult, error)`:

```go
type StartRunResult struct {
    Run       *RunView       `json:"run,omitempty"`
    Bootstrap *BootstrapHint `json:"bootstrap,omitempty"`
}

type BootstrapHint struct {
    Reason string `json:"reason"`  // "manifest_missing" hoje; futuro: "schema_outdated", etc.
}
```

UI:
- `result.Run != nil` → fluxo normal (mostra ▶ Stop, Logs, etc.)
- `result.Bootstrap != nil` → mostra BootstrapModal

Vantagens vs sentinel error string parsing: tipado, extensível, sem regex frágil.

### Evento `bootstrap.proposed`

Mantém o nome (não quebra `useSessionEvents.test.ts:219`); estende payload. O
struct Go usa JSON tags snake_case explícitas — `events.LazyEmitter.Emit`
serializa via padrão `json.Marshal`, então sem tags os nomes viriam CamelCase
e o tipo TS não casaria.

```go
// internal/core/bootstrap.go
type BootstrapProposedPayload struct {
    TaskID       string   `json:"task_id"`
    SessionID    string   `json:"session_id"`
    ManifestText string   `json:"manifest_text"`
    Valid        bool     `json:"valid"`
    Errors       []string `json:"errors"`        // [] não nil; preencher com `make([]string, 0)` se válido
}
```

```typescript
// ui/src/lib/events.ts
| { type: 'bootstrap.proposed';
    task_id: string;            // era null; passa a ser string
    session_id: string;
    manifest_text: string;      // YAML lido do disco
    valid: boolean;
    errors: string[];           // [] se valid=true; mensagens human-readable se false
  }
```

## Tratamento de erros

| Cenário | Comportamento |
|---|---|
| `ErrTaskInTerminalState` | Reject. |
| `ErrSandboxUnavailable` (ai-jail/terminal missing) | Reject. |
| **`ErrManifestAlreadyExists`** (novo) | `<cwd>/.orchestrator/run.yml` já existe no Start → reject; UI traduz pra "manifest apareceu, clique Run de novo". |
| Worktree creation falha | Propaga sem cleanup (nada foi escrito). |
| `os.MkdirAll` falha | Propaga. |
| Escrita do BOOTSTRAP_PROMPT.md falha | Propaga. |
| Escrita do `.ai-jail` falha | Propaga + remove BOOTSTRAP_PROMPT.md. |
| `fsnotify.NewWatcher` falha | Propaga + remove arquivos. |
| `runtime.Spawn` falha | Propaga + remove arquivos + watcher.Close(). |
| `os.ReadFile` no handleDetection falha | Log + continua escutando. |
| `LoadManifest` falha | Emit `{valid:false, errors}`. Entry, watcher, runtime, prompt todos persistem. |
| Watcher.Errors channel emite | Log + continua. |
| `runtime.Kill` falha durante cleanup | Log + segue. |

### Idempotência

- `Start(taskID)` 2x → mesma BootstrapView; spawn chamado 1x.
- `Cancel(taskID)` 2x → 1º cleanup; 2º no-op.
- Watcher detecta o mesmo arquivo 3x (eventos duplicados do fsnotify) → 1º
  cleanup; 2º+3º veem `!ok` e desistem.

### Recuperação pós-crash da app

Sem persistência (efêmero por design). Reinício perde o map. Claude detached
sobrevive até usuário fechar o terminal. Próximo Start no mesmo task sobrescreve
arquivos idempotentemente. **Documentado como limitação aceita** — Python
original também não tinha recovery. YAGNI.

- `fakeRuntime` (já existe em `internal/core/sessions_test.go`) — reusar.

## Migração de dados

Nenhuma. Bootstrap é efêmero — zero DB.

## Carregamento (entrega)

Spec #2 entrega valor incremental sobre #1:
- #1 sozinho: Run funciona em projetos que **já têm** manifest commitado.
- #1 + #2: Run funciona em qualquer projeto Go/Node/Python comum em ~30s
  (tempo do Claude bootstrap ler README + escrever YAML).

## Dependências externas

- `github.com/fsnotify/fsnotify v1.x` — única adição em `go.mod`. Stable,
  pacote canônico, usado por Docker/k8s. Funciona em Linux via inotify; macOS
  via FSEvents; Windows via ReadDirectoryChangesW. Linux é o único alvo agora.

## Não-objetivos confirmados

- **Sem polling fallback** (decisão #2). fsnotify falhar é tratado como erro
  fatal no Start; usuário fecha modal e reabre.
- **Sem rate-limit em handleDetection.** fsnotify duplica eventos sob save com
  rename atômico, mas o critical-section-claim cobre. Não precisa de timer.
- **Sem auto-cancel quando UI fecha sem clicar Cancel.** Watcher continua até
  app desligar OU usuário discartar a task (cleanup chain).
- **Sem persistência do prompt template fora do binário.** `BOOTSTRAP_PROMPT.md`
  é embedado, não configurável por usuário. Customização vira F10.7+.

## Riscos

| Risco | Mitigação |
|---|---|
| Claude bootstrap escreve manifest em local errado (não `<cwd>/.orchestrator/run.yml`) | `BOOTSTRAP_PROMPT.md` instrui caminho exato; cmd `pwd` no terminal mostra cwd correto |
| Manifest gerado é válido sintaticamente mas semanticamente errado (imagem inexistente, etc.) | RunsService.StartRun vai falhar na build/pull com erro de Docker; spec #1 trata rollback |
| fsnotify perde evento sob load extremo | Aceito; usuário pode forçar com `touch run.yml` ou re-rodar bootstrap |
| Múltiplos saves rápidos durante Claude escrevendo via Write tool | Critical-section-claim cobre; 1 cleanup, eventos extras descartados |
| Watcher goroutine vazada se cancel não for chamado | `defer watcher.Close()` na goroutine + `ctx.Done()` no `select` garante saída limpa |
