## 1. Contexto

A feature "Run from Panel" permite o usuГЎrio clicar um botГЈo в–¶ na task e ter
um stack Docker subindo localmente (DB + backend + frontend, ou qualquer set
de services descrito num manifesto YAML). Quando os services ficam healthy,

O Python `orchestrator/` jГЎ tinha isso (~1000 LOC entre `core/runs.py`,
`core/manifest.py`, `core/port_allocator.py`, `sandbox/docker_ops.py`,
`api/runs.py`). Esta spec porta verbatim a semГўntica.

## 2. DecisГµes de design

| # | DecisГЈo | RazГЈo |
|---|---|---|
| 1 | Duas specs (happy path + bootstrap) | Spec #1 sozinha jГЎ Г© entregГЎvel demonstrГЎvel. Bootstrap (sessГЈo Claude + file watcher) Г© opcional/condicional. |
| 2 | Subprocess via `os/exec` (nГЈo Docker SDK Go) | Mesma arquitetura do Python. SDK Go (`github.com/docker/docker/client`) Г© dep pesada (~30 MB transitivo). Pra up/down/logs subprocess Г© suficiente. |
| 3 | DockerOps como interface + SubprocessDockerOps impl | Testabilidade. `FakeDockerOps` permite unit tests da `RunsService` sem fork de Docker. Integration test atrГЎs de `//go:build integration`. |
| 4 | Port range 31000-31999 fixo | Mesmo do Python (ADR-0005 original). 1000 ports cobre N runs simultГўneos. |
| 5 | Manifest path `.orchestrator/run.yml` (committed) | Mesmo do Python. Lido a cada Start (sem cache вЂ” manifest pode mudar). |
| 6 | State machine `pendingв†’buildingв†’seedingв†’ready` / `в†’failed` / `в†’stopped` | Mesmo do Python (В§8 do F6 spec). |
| 7 | 1 run ativa por task (partial unique index) | DB enforces; service tambГ©m checa antes (early error). |
| 8 | 3-layer rollback em qualquer erro pГіs-create | Containers stop+rm в†’ network rm в†’ ports release в†’ marca DB failed. Best-effort em cada layer. |
| 9 | Boot-time orphan cleanup | App pode crashar com runs ativos; cleanup mata containers + libera ports + marca runs `stopped` com reason `orphan_cleanup_on_boot`. |
| 10 | Runs cleanup callback no `core.TasksService.Patch` | Quando task entra em estado terminal: ordem **sessions в†’ runs в†’ worktrees**. Containers podem ter volumes do worktree, entГЈo runs antes de worktree cleanup. |
| 11 | SSE logs endpoint mountado em `localhttp.Server` | Reusa o listener compartilhado de F10.5; sem auth (loopback). |
| 12 | Sem `multi-arch buildx`, sem hot-reload | YAGNI; containers que querem hot-reload jГЎ lidam internamente (mount + nodemon/air). |
| 13 | Sem migration nova nesta spec | `run_instances` table jГЎ existe. |

## 4. Manifest

### 4.1 Path

Lido a cada `StartRun` a partir de `<run-cwd>/.orchestrator/run.yml`.
Run-cwd derivado por В§6.

### 4.2 Schema (Go types, mirror Python verbatim)

```go
type HealthcheckSpec struct {
    Test     []string      `yaml:"test"     json:"test"`     // ex: ["CMD", "pg_isready"]
    Interval time.Duration `yaml:"interval" json:"interval"` // default 5s
    Timeout  time.Duration `yaml:"timeout"  json:"timeout"`  // default 3s
    Retries  int           `yaml:"retries"  json:"retries"`  // default 10
}

    Command []string      `yaml:"command" json:"command"`     // ex: ["psql", "-f", "/seeds/init.sql"]
    Timeout time.Duration `yaml:"timeout" json:"timeout"`     // default 60s
}

type ServiceSpec struct {
    Image       string            `yaml:"image,omitempty"       json:"image,omitempty"`
    Build       string            `yaml:"build,omitempty"       json:"build,omitempty"`        // path relative to cwd
    Port        int               `yaml:"port,omitempty"        json:"port,omitempty"`         // container port; 0 = no host expose
    DependsOn   []string          `yaml:"depends_on,omitempty"  json:"depends_on,omitempty"`
    Env         map[string]string `yaml:"env,omitempty"         json:"env,omitempty"`         // values support $PORT_<svc>, $URL_<svc>, $RUN_ID, $CWD
    Healthcheck *HealthcheckSpec  `yaml:"healthcheck,omitempty" json:"healthcheck,omitempty"`
    Seed        *SeedSpec         `yaml:"seed,omitempty"        json:"seed,omitempty"`

}

type ManifestSpec struct {
    Version  string                 `yaml:"version"  json:"version"`   // "1"
    Services map[string]ServiceSpec `yaml:"services" json:"services"`
}
```

### 4.3 ValidaГ§ГЈo (estГЎtica, antes de Allocate)

Regras (mesmas do Python):

1. `version == "1"` (literal).
2. Cada `service` tem `image` XOR `build` (exatamente um).
3. `depends_on` referencia services existentes.
4. Sem ciclos em `depends_on` (DFS detecГ§ГЈo).
5. Sem duplicate ports entre services (cada um pega um host port diferente вЂ” port allocator garante isso).
6. **Token-graph vГЎlido**: qualquer `$PORT_<svc>` ou `$URL_<svc>` em values
   de `env` referencia um service que (a) existe no manifest e (b) tem
   `port:` definido. ValidaГ§ГЈo **estГЎtica**, parte do `Validate()`, antes
   de qualquer allocate.

Erros agregados em uma mensagem sГі (mesmo padrГЈo do `catalog.Load`).

### 4.4 SubstituiГ§Гµes (runtime, depois de Allocate)

`ResolveSubstitutions(env, ports, runID, cwd)` aplica em valores de `env`
(nГЈo em chaves, nГЈo em outros campos):

| Token | SubstituГ­do por |
|---|---|
| `$PORT_<svc>` | host port alocado para `<svc>` (int в†’ string) |
| `$URL_<svc>` | `http://localhost:<host_port>` |
| `$RUN_ID` | UUID do run |
| `$CWD` | path absoluto do run-cwd |

SubstituiГ§ГЈo Г© **infalГ­vel em runtime** вЂ” tokens jГЎ foram validados
estaticamente em В§4.3 regra #6 contra o grafo de services + presence de
`port:`. Por construГ§ГЈo, todos os `$PORT_<svc>` e `$URL_<svc>` resolvem.

## 5. Port allocator

### 5.1 Shape

```go
type PortAllocator struct {
    mu       sync.Mutex
    reserved map[int]bool
}

func NewPortAllocator() *PortAllocator

func (p *PortAllocator) Allocate() (int, error)
func (p *PortAllocator) Release(port int)
func (p *PortAllocator) Reserve(port int)
```

### 5.2 Range

Fixo **31000-31999** (1000 ports). Constantes `MinPort = 31000`, `MaxPort = 31999`.

### 5.3 Allocate

Varre `[MinPort, MaxPort]` em ordem:
1. Skip se `reserved[port] == true`.
2. Probe: `ln, err := net.Listen("tcp", "127.0.0.1:<port>")`; se erro, port estГЎ em uso por outro processo вЂ” skip. Se sucesso, fecha imediato e o port Г© livre.
3. Marca `reserved[port] = true` e retorna.

Se todos os 1000 ports estГЈo ocupados (`reserved` cheio ou usados por outros processos), retorna `ErrNoFreePort`.

### 5.4 PersistГЄncia

`reserved` Г© in-memory only. Recuperado no boot via `main.go` (ver В§3.3
para o snippet canГґnico com error handling; o trecho abaixo Г©
ilustrativo):

```go
// snippet ilustrativo вЂ” wiring real estГЎ em В§3.3
for _, run := range active {
    for _, port := range run.Ports() {
        portAlloc.Reserve(port)
    }
}
```

Boot orphan cleanup (В§7.4) chama `Release` pra cada port quando finaliza run ГіrfГЈo вЂ” entГЈo em estado estГЎvel post-boot, `reserved` reflete sГі runs **ativos legГ­timos**.

### 7.2 StartRun flow

```go
func (s *RunsService) StartRun(ctx context.Context, taskID string) (*store.Run, error) {
    // 1. Preflight: docker disponГ­vel
    if _, err := exec.LookPath("docker"); err != nil {
        return nil, ErrDockerUnavailable
    }

    // 2. Task guards
    task, err := s.tasks.Get(ctx, taskID)
    if err != nil { return nil, err }
    if IsTerminal(task.State) { return nil, ErrTaskNotEligible }

    // 3. Run guard (1 active per task; partial unique index also enforces)
    if existing, _ := s.repo.GetActiveByTask(ctx, taskID); existing != nil {
        return nil, ErrTaskHasActiveRun
    }

    // 4. Derive CWD
    cwd, err := s.DeriveRunCWD(ctx, taskID)
    if err != nil { return nil, err }

    // 5. Load + validate manifest
    manifest, err := LoadManifest(cwd)
    if err != nil { return nil, err }  // ErrManifestMissing or ErrManifestInvalid
    if err := manifest.Validate(); err != nil { return nil, fmt.Errorf("%w: %v", ErrManifestInvalid, err) }

    // 6. Topo sort
    order, err := topoSort(manifest.Services)
    if err != nil { return nil, err }  // ErrCircularDeps

    // 7. Allocate ports
    ports := map[string]int{}
    var allocated []int
    for _, svc := range order {
        spec := manifest.Services[svc]
        if spec.Port == 0 { continue }
        p, err := s.allocator.Allocate()
        if err != nil {
            for _, alloc := range allocated { s.allocator.Release(alloc) }
            return nil, err
        }
        ports[svc] = p
        allocated = append(allocated, p)
    }

    // 8. Resolve substitutions per service
    runID := uuid.NewString()
    for svc, spec := range manifest.Services {
        manifest.Services[svc] = applySubst(spec, ports, runID, cwd)
    }

    // 9. Insert run row (status=pending)
    run := store.Run{
        ID: runID, TaskID: taskID, Status: "pending",
        Cwd: cwd, ManifestPath: filepath.Join(cwd, ".orchestrator/run.yml"),
        PortsJSON: marshalPorts(ports),
        NetworkName: "jarvis-run-" + runID[:8],
        StartedAt: time.Now(),
    }
    if err := s.repo.Insert(ctx, run); err != nil {
        for _, alloc := range allocated { s.allocator.Release(alloc) }
        return nil, err  // row was never committed; no DB cleanup needed
    }
    s.emitStatus(&run, "", "pending")

    // 10. Build images
    s.transition(&run, "building")
    for _, svc := range order {
        spec := manifest.Services[svc]
        if spec.Build == "" { continue }
        tag := "jarvis-run-" + runID[:8] + "-" + svc
        if err := s.docker.Build(ctx, filepath.Join(cwd, spec.Build), tag); err != nil {
            s.rollback(ctx, &run, nil, run.NetworkName, allocated, err.Error())
            return nil, err
        }
    }

    // 11. Network create
    if err := s.docker.NetworkCreate(ctx, run.NetworkName); err != nil {
        s.rollback(ctx, &run, nil, "", allocated, err.Error())
        return nil, err
    }

    // 12. Container starts (in order)
    containerIDs := map[string]string{}
    for _, svc := range order {
        spec := manifest.Services[svc]
        cs := buildContainerSpec(svc, spec, run, ports[svc])
        cid, err := s.docker.ContainerStart(ctx, cs)
        if err != nil {
            s.rollback(ctx, &run, containerIDs, run.NetworkName, allocated, err.Error())
            return nil, err
        }
        containerIDs[svc] = cid

        // Healthcheck wait per-service before starting dependents
        if spec.Healthcheck != nil {
            if err := s.waitHealthy(ctx, cid, spec.Healthcheck); err != nil {
                s.rollback(ctx, &run, containerIDs, run.NetworkName, allocated, "healthcheck timeout: "+svc)
                return nil, err
            }
        }
    }

    // 13. Seeds
    s.transition(&run, "seeding")
    for _, svc := range order {
        spec := manifest.Services[svc]
        if spec.Seed == nil { continue }
        if err := s.docker.RunInContainer(ctx, containerIDs[svc], spec.Seed.Command, spec.Seed.Timeout); err != nil {
            s.rollback(ctx, &run, containerIDs, run.NetworkName, allocated, "seed failed: "+svc+": "+err.Error())
            return nil, err
        }
    }

    // 14. Persist final state
    run.ContainersJSON = marshalContainers(containerIDs)
    s.transition(&run, "ready")

    return &run, nil
}
```

### 7.3 rollback (3-layer)

```go
func (s *RunsService) rollback(ctx context.Context, run *store.Run,
    containerIDs map[string]string, networkName string,
    ports []int, errMsg string,
) {
    // Layer 1: containers (best-effort)
    for _, cid := range containerIDs {
        _ = s.docker.Stop(ctx, cid)
        _ = s.docker.Rm(ctx, cid)
    }
    // Layer 2: network
    if networkName != "" {
        _ = s.docker.NetworkRm(ctx, networkName)
    }
    // Layer 3: ports
    for _, p := range ports {
        s.allocator.Release(p)
    }
    // DB: mark failed
    _ = s.repo.MarkFailed(ctx, run.ID, errMsg)
    s.emitStatus(run, run.Status, "failed")
}
```

### 7.4 Boot orphan cleanup

```go
func (s *RunsService) CleanupOrphans(ctx context.Context) error {
    active, err := s.repo.ListActive(ctx)
    if err != nil { return err }
    for _, run := range active {
        for svc, cid := range run.ContainerIDs() {
            _ = s.docker.Stop(ctx, cid)
            _ = s.docker.Rm(ctx, cid)
            log.Printf("orphan cleanup: run %s service %s container %s", run.ID, svc, cid)
        }
        if run.NetworkName != "" {
            _ = s.docker.NetworkRm(ctx, run.NetworkName)
        }
        _ = s.repo.MarkEnded(ctx, run.ID, "stopped", "orphan_cleanup_on_boot")

    return nil
}
```

### 7.5 CleanupForTask (task terminal callback)

```go
func (s *RunsService) CleanupForTask(ctx context.Context, taskID string) error {
    run, _ := s.repo.GetActiveByTask(ctx, taskID)
    if run == nil { return nil }
    return s.StopRun(ctx, run.ID)  // graceful stop, same path as user click
}
```

`tasksSvc.Patch` chama na ordem **sessions в†’ runs в†’ worktrees** quando
atinge estado terminal (В§4.3).

**Container-stop paralelizaГ§ГЈo dentro de uma run.** `docker stop --time 10`
demora atГ© 10s por container. Uma run com 5 containers sequencialmente
bloquearia `Patch` por ~50s. SoluГ§ГЈo: dentro de uma Гєnica `StopRun`,
`Stop` Г© disparado **em paralelo por service** (goroutines + sync.WaitGroup).
O caller observa sГі o `max(stop_durations)`, nГЈo a soma.

Sketch do `StopRun`:

```go
func (s *RunsService) StopRun(ctx context.Context, runID string) error {
    run, err := s.repo.GetByID(ctx, runID)
    if err != nil { return err }
    if run.EndedAt != nil { return nil }  // idempotente

    s.transition(run, "stopping")

    // Stop containers em paralelo
    var wg sync.WaitGroup
    for _, cid := range run.ContainerIDs() {
        wg.Add(1)
        go func(id string) {
            defer wg.Done()
            _ = s.docker.Stop(ctx, id)
            _ = s.docker.Rm(ctx, id)
        }(cid)
    }
    wg.Wait()

    // Network + ports apГіs todos containers down
    _ = s.docker.NetworkRm(ctx, run.NetworkName)
    for _, port := range run.Ports() {
        s.allocator.Release(port)
    }

    return s.repo.MarkEnded(ctx, run.ID, "stopped", "")
}
```

`CleanupForTask` em si fica **sГ­ncrono** (chama uma Гєnica `StopRun`).
O Patch terminal nГЈo fica fire-and-forget porque o caller (UI/MCP) precisa
saber quando o cleanup completou pra refletir o estado final visualmente.

### 8.2 SubprocessDockerOps

Cada mГ©todo Г© um `exec.Command("docker", args...)`:

- `Build`: `docker build -t <tag> <path>`
- `NetworkCreate`: `docker network create --driver bridge <name>`
- `NetworkRm`: `docker network rm <name>`
- `ContainerStart`: `docker run -d --name <name> --network <net> --network-alias <alias> -p <hp>:<cp> -e K=V -v <hp>:<cp> <image>`. Returns container ID from stdout.
- `RunInContainer`: `docker exec <id> <command...>` with `context.WithTimeout(timeout)`
- `StreamLogs`: `docker logs -f <id>` вЂ” `cmd.StdoutPipe()` в†’ caller writes to dst via bufio.Scanner. Process killed when ctx cancels.
- `Stop`: `docker stop --time 10 <id>` (10s graceful, then SIGKILL)
- `Rm`: `docker rm -f <id>`
- `ContainerHealthStatus`: `docker inspect --format='{{.State.Health.Status}}' <id>` вЂ” returns "starting", "healthy", "unhealthy", or empty string (no healthcheck defined).

### 8.3 waitHealthy

```go
func (s *RunsService) waitHealthy(ctx context.Context, cid string, hc *HealthcheckSpec) error {
    deadline := time.Now().Add(hc.Interval * time.Duration(hc.Retries))
    for time.Now().Before(deadline) {
        select {
        case <-ctx.Done():
            return ctx.Err()
        default:
        }
        status, err := s.docker.ContainerHealthStatus(ctx, cid)
        if err == nil && status == "healthy" {
            return nil
        }
        time.Sleep(hc.Interval)
    }
    return fmt.Errorf("healthcheck timeout after %v", hc.Interval*time.Duration(hc.Retries))
}
```

### 9.2 SSE logs (mounted on localhttp)

`GET /api/runs/{runID}/logs?service=<name>` в†’ SSE stream:

```
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive

data: line 1 of container stdout
data: line 2

...
```

Implementation:
```go
func (a *RunsAPI) LogsHandler() http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        runID := r.PathValue("runID")
        svc := r.URL.Query().Get("service")

        w.Header().Set("Content-Type", "text/event-stream")
        w.Header().Set("Cache-Control", "no-cache")
        flusher, _ := w.(http.Flusher)

        cid, err := a.svc.ContainerIDFor(r.Context(), runID, svc)
        if err != nil {
            http.Error(w, err.Error(), http.StatusNotFound)
            return
        }

        pr, pw := io.Pipe()
        defer pr.Close()  // unblocks the goroutine if scanner errors first
        go func() {
            defer pw.Close()
            _ = a.svc.StreamLogs(r.Context(), cid, pw)
        }()

        scanner := bufio.NewScanner(pr)
        for scanner.Scan() {
            fmt.Fprintf(w, "data: %s\n\n", scanner.Text())
            if flusher != nil { flusher.Flush() }
            if r.Context().Err() != nil { return }
        }
    })
}
```

**Cleanup garantido em disconnect:** quando o HTTP client desconecta,
`r.Context()` cancela em cascata: (1) `StreamLogs` propaga ctx pro
`docker logs -f` subprocess via `exec.CommandContext`, que recebe SIGTERM,
(2) o write em `pw` falha com pipe broken, (3) `pw.Close()` no defer,
(4) `pr.Read` no scanner retorna EOF, (5) handler exit + `defer pr.Close()`.
Sem goroutines vazadas, sem processo zumbi.

Mountado: `localSrv.Mount("/api/runs/", runsAPI.LogsHandler())`. UI consome via
`EventSource(\`${LOCAL_HTTP_BASE}/api/runs/${runID}/logs?service=${svc}\`)`
(jГЎ scaffoldado em `ui/src/lib/runSseClient.ts`).
